    51→### 2. Layer Separation from Depth Maps
    52→
    53→Given a continuous grayscale depth map, splitting into N discrete layers:
    54→
    55→**a) Equal-width thresholding (simplest)**: normalize depth to [0,1], slice into N bands. Easy but tends to over-segment smooth gradients.
    56→
    57→**b) K-means / quantile clustering** (recommended for 3–5 layers): cluster depth pixels into K bins using `sklearn.cluster.KMeans` or 1-D k-means on the depth histogram. Adapts to actual scene depth distribution. Reference: "Object-Background Region Separation in Depth Image using k-Means Clustering" (KISM 2021).
    58→
    59→**c) Bilateral filtering first** (used by Shih et al. 3D-Photo-Inpainting): edge-preserving smoothing of the depth map before clustering → cleaner layer boundaries that align with image edges.
    60→
    61→**d) Watershed / connected-component labeling**: produce spatially coherent regions (not just depth bands) → useful for "subject vs background" rather than "near vs far".
    62→
    63→**e) Adaptive layering (Layered Depth Image, LDI)**: the gold standard from Shih et al. — splits along depth *discontinuities* with explicit pixel connectivity, producing a graph of front-to-back layers each with its own inpainted color+depth. This is what 3D Photo Inpainting / Immersity use.
    64→
    65→For our app, **b + c (bilateral-filter then 1-D k-means with K=3–5)** is the practical sweet spot for "foreground / midground / background / sky" decomposition. Save the cluster centroids as depth thresholds and emit per-layer masks as PNGs.
    66→
    67→### 3. Segmentation-Based Separation (SAM / SAM2)
    68→
    69→**Meta SAM2** (`facebookresearch/sam2`, Apache 2.0) is the SOTA. Two modes:
    70→- **Promptable**: click/box/mask → precise mask. Great for "user clicks subject".
    71→- **Automatic mask generator**: produces all plausible masks. Then we filter by area / centrality to pick the main subject.
    72→
    73→**Sky detection**: dedicated models exist (SkyFinder dataset, "Segmenting Sky Pixels in Images"), but a simpler heuristic often suffices — combine (a) top-of-image region, (b) blue-ish / bright color, (c) **far depth** (depth value in the top decile). Can also use a semantic seg model like **ADE20K-trained OneFormer / Mask2Former** which has a "sky" class out of the box.
    74→
    75→**Ground detection**: similar — bottom-of-image + dark/uniform color + near depth. Or ADE20K class "floor / grass / road / earth".
    76→
    77→**Subject detection**: SAM2 auto-mask → pick the largest non-sky/non-ground mask whose centroid is in the central 60% of the frame. Optionally refine edges with **Matting-Anything** or **MODNet** for hair-level fidelity (important for portraits).
    78→
    79→**Recommended pipeline**:
    80→1. Run ADE20K semantic seg (1 forward pass) → labels every pixel as sky / ground / person / building / etc.
    81→2. Run SAM2 auto-mask → instance-level subject candidates.
    82→3. Cross-reference: subject = SAM2 mask overlapping "person" or central non-background ADE20K labels.
    83→
    84→For a Next.js app, run server-side via the `transformers.js` SAM2 port or a Python microservice. **transformers.js** can run SAM2 in Node.js directly — no Python needed.

    86→### 4. Background Removal + Inpainting
    87→
    88→**Background removal**:
    89→- **Rembg** (Python, MIT) — wraps U2Net / BRIA RMBG / IS-Net. CLI + library. Production-grade, ~1–2s per image on CPU. Best free option.
    90→- **@imgly/background-removal** (JS, runs in browser or Node via WASM) — drop-in for Next.js, no Python.
    91→
    92→**Inpainting the occluded background** (critical for parallax quality):
    93→- **LaMa** (Resolution-robust Large Mask Inpainting with Fourier Convolutions, Samsung Research) — SOTA for large mask inpainting, generalizes to 2K res despite 256×256 training. Fast (~1s/image on GPU). Apache 2.0.
    94→- **IOPaint** (formerly Lama-Cleaner, MIT) — self-hostable web app + Python lib + HTTP API that wraps LaMa, Stable Diffusion inpainting, MAT, ZITS, FcF, Manga, PaintByExample, and several matting models. **This is the swiss-army knife.** Docker container → Next.js calls `POST http://iopaint:8000/inpaint` with image + mask.
    95→- **Stable Diffusion Inpainting** — highest quality for complex scenes, but slow (5–15s on GPU) and may "hallucinate" extra content.
    96→- **3D Photo Inpainting** (Shih et al., CVPR 2020, `vt-vl-lab/3d-photo-inpainting`) — the canonical method. Takes RGB+depth, outputs a Layered Depth Image (LDI) with color+depth for occluded regions. Heavy (~10 min on A100 per image via Replicate). Available as `pollinations/3d-photo-inpainting` on Replicate. **Best quality**, expensive.
    97→- **SLIDE** (ICCV 2021) — soft layering variant; handles hair/foliage better. No official code; community reimplementations exist.
    98→
    99→**Practical recommendation for our app**:
   100→1. Default: Rembg (or `@imgly/background-removal`) for subject alpha matte.
   101→2. **Dilate the subject mask by ~20–40 px** (this is what Facebook 3D Photos / Tiefling's `expandDepthmapRadius` do — exposes a band of background that can be revealed when subject shifts).
   102→3. Run **LaMa via IOPaint** on the dilated mask → reconstructs background where the subject used to be. Store this as `background_inpainted.png`.
   103→4. For premium mode: route through `pollinations/3d-photo-inpainting` on Replicate for the full LDI treatment (cached, async).

   149→### 7. The "LSD Alive" Effect — Subtle Animation Toolkit
   150→
   151→The "alive" feeling is the *combination* of several subtle, low-amplitude animations. Each alone is barely noticeable; together they create the magic. Key techniques (ranked by impact):
   152→
   153→1. **Mouse / gyro parallax** (the foundation). Map `mouseX/Y` (or `deviceorientation`) to camera/layer offset, multiplied by per-layer depth. Smooth with lerp (damping ~0.05–0.1).
   154→2. **Breathing / pulsing** — slow (4–8s period) `scale: 1.0 ↔ 1.015`. Apply to whole image and slightly stronger to foreground.
   155→3. **Sway** — gentle `rotate: ±0.3°` oscillation on a 6–10s sine. Subtle but very "alive".
   156→4. **Liquid / wavy distortion** — `<feTurbulence>` SVG filter animated via `seed` or `baseFrequency`, applied with low `scale` (1–3px displacement). Gives the "water surface" shimmer. Optional per-region.
   157→5. **Floating particles** — CSS / canvas particles (dust motes, light specks) drifting upward or with mouse. ~10–30 particles, low opacity.
   158→6. **Light shimmer** — slow gradient overlay sweeping across (radial highlight following mouse, or auto-cycling). ~2–5% opacity.
   159→7. **Chromatic aberration / RGB split** — shift R/B channels 1–3px based on mouse distance from center. Increases at edges (radial).
   160→8. **Depth-of-field shift** — blur the far layer (or near layer) alternately on a 8–12s cycle. `filter: blur(0–4px)`.
   161→9. **Ken Burns** — slow zoom (1.0 → 1.05 over 20–30s) + sub-pixel pan. Good as a base when there's no mouse input.
   162→10. **Vignette pulse** — very subtle darkening at edges pulsing with breathing.
   163→
   164→Amplitude guidelines: any single effect should be barely perceptible when stared at. Combined, they shouldn't draw attention to themselves. The user should just feel "the image is breathing".

   166→### 8. Code Examples
   167→
   168→**a) Codrops / canonical WebGL fragment shader** (the foundation of LeiaPix/DepthFlow/Tiefling):
   169→```glsl
   170→// Fragment shader: original image + depth map → parallax with mouse
   171→uniform sampler2D originalImage;
   172→uniform sampler2D depthImage;
   173→uniform vec2 mouse;  // normalized -1..1, damped
   174→uniform float depthStrength;  // ~0.02 for subtle
   175→varying vec2 vUv;
   176→
   177→void main() {
   178→    vec4 depth = texture2D(depthImage, vUv);
   179→    // White = close → moves more with mouse; black = far → stays put
   180→    vec2 offset = mouse * depth.r * depthStrength;
   181→    gl_FragColor = texture2D(originalImage, vUv + offset);
   182→}
   183→```
   184→That's the whole effect. ~6 lines of GLSL. From the Codrops tutorial by Yuri Artiukh.
   185→
   186→**b) React Three Fiber implementation** (R3F + drei):
   187→```tsx
   188→import { Canvas, useFrame, useLoader } from '@react-three/fiber'
   189→import { TextureLoader } from 'three'
   190→import { useRef, useMemo } from 'react'
   191→import * as THREE from 'three'
   192→
   193→const vertexShader = `
   194→  varying vec2 vUv;
   195→  void main() { vUv = uv; gl_Position = vec4(position, 1.0); }
   196→`
   197→const fragmentShader = `
   198→  uniform sampler2D uImage; uniform sampler2D uDepth;
   199→  uniform vec2 uMouse; uniform float uTime;
   200→  varying vec2 vUv;
   201→  void main() {
   202→    vec4 d = texture2D(uDepth, vUv);
   203→    // subtle breathing + mouse parallax
   204→    float breath = sin(uTime * 0.8) * 0.003 * d.r;
   205→    vec2 off = uMouse * d.r * 0.03 + vec2(breath, breath * 0.5);
   206→    gl_FragColor = texture2D(uImage, vUv + off);
   207→  }
   208→`
   209→
   210→function AlivePlane({ img, depth }) {
   211→  const mat = useRef<THREE.ShaderMaterial>(null!)
   212→  const tex = useLoader(TextureLoader, img)
   213→  const depthTex = useLoader(TextureLoader, depth)
   214→  const mouse = useMemo(() => new THREE.Vector2(0, 0), [])
   215→  useFrame((state, delta) => {
   216→    // smooth mouse follow
   217→    mouse.x += (state.pointer.x - mouse.x) * 0.05
   218→    mouse.y += (state.pointer.y - mouse.y) * 0.05
   219→    mat.current.uniforms.uMouse.value = mouse
   220→    mat.current.uniforms.uTime.value += delta
   221→  })
   222→  return (
   223→    <mesh>
   224→      <planeGeometry args={[2, 2]} />
   225→      <shaderMaterial ref={mat} vertexShader={vertexShader} fragmentShader={fragmentShader}
   226→        uniforms={{
   227→          uImage:  { value: tex },
   228→          uDepth:  { value: depthTex },
   229→          uMouse:  { value: mouse },
   230→          uTime:   { value: 0 },
   231→        }} />
   232→    </mesh>
   233→  )
   234→}
   235→```

   312→### 9. Depth-Map-to-Displacement Technique — Deep Dive (the LeiaPix/Immersity method)
   313→
   314→This is the single most important technique for our project. Two rendering variants:
   315→
   316→**Variant A — Fragment-shader UV offset** (Codrops, Depthy, Pixi DisplacementFilter):
   317→- 1 fullscreen quad, fragment shader samples depth, offsets UV.
   318→- ✅ Trivial to implement, ~6 lines GLSL, runs at 60fps on any GPU.
   319→- ❌ Produces **"rubber band" / stretch artifacts** at depth discontinuities: when a foreground pixel shifts, it leaves a hole filled by the next pixel's color → smeared edges.
   320→
   321→**Variant B — Vertex-displaced mesh** (Tiefling, three.js forum examples):
   322→- Subdivided plane geometry; each vertex's Z = depth value.
   323→- ✅ True parallax geometry, handles occlusion correctly via depth buffer.
   324→- ❌ Mesh resolution tradeoff: vertex per pixel = too many verts (2M for 1080p). Need adaptive meshing.
   325→
   326→**Variant C — Layered / multi-plane** (Facebook 3D Photo, Immersity, 3D-Photo-Inpainting):
   327→- N depth planes, each with its own (potentially inpainted) color texture.
   328→- ✅ Best quality: occlusion solved by inpainting behind each foreground layer.
   329→- ❌ Requires per-layer background inpainting; more complex asset pipeline.
   330→
   331→**The key quality unlocks (do ALL of these)**:
   332→1. **Dilate the depth map at object edges** before using as displacement (Tiefling's `expandDepthmapRadius`). This pre-extends the background so when foreground shifts, there's real background to show.
   333→2. **Inpaint the background** behind the main subject (LaMa). Ship `background_inpainted.png` as a separate texture sampled where `depth < subject_depth`.
   334→3. **Limit motion range** (Google Cinematic Photos insight): clamp mouse offset to ±10° or ±30–50 px. Beyond that, artifacts become visible.
   335→4. **Use easing / damping** on mouse so motion never exceeds comfortable range.
   336→5. **Apply bilateral filtering** to the depth map to remove noisy speckles before slicing.
   337→6. For best results, render with **variant B (vertex mesh) using the inpainted background as a second layer behind** — gets ~90% of Immersity quality at 10% of the compute.
   338→
   339→**Concrete shader with occlusion-aware background fill** (Variant C, simplified):
   340→```glsl
   341→uniform sampler2D uForeground;   // original image
   342→uniform sampler2D uBackground;   // LaMa-inpainted background (subject removed)
   343→uniform sampler2D uDepth;
   344→uniform sampler2D uSubjectMask;  // 1=subject, 0=background
   345→uniform vec2 uMouse;
   346→uniform float uTime;
   347→varying vec2 vUv;
   348→
   349→void main() {
   350→  vec4 d = texture2D(uDepth, vUv);
   351→  float mask = texture2D(uSubjectMask, vUv).r;
   352→  float breath = sin(uTime * 0.6) * 0.002;
   353→
   354→  // Foreground shifts more (subject), background less
   355→  vec2 bgOff = uMouse * (1.0 - d.r) * 0.015 + vec2(breath);
   356→  vec2 fgOff = uMouse * d.r * 0.05 + vec2(breath * 2.0);
   357→
   358→  vec3 bg = texture2D(uBackground, vUv + bgOff).rgb;
   359→  vec3 fg = texture2D(uForeground, vUv + fgOff).rgb;
   360→
   361→  // Use the displaced mask to decide which layer shows
   362→  float displacedMask = texture2D(uSubjectMask, vUv + fgOff).r;
   363→  vec3 color = mix(bg, fg, displacedMask);
   364→
   365→  gl_FragColor = vec4(color, 1.0);
   366→}
   367→```

   373→### 🥇 Rank 1 — Hybrid: Depth Anything V2 + LaMa + Three.js shader (RECOMMENDED)
   374→**Quality: 9/10 | Feasibility: 8/10 | Effort: Medium | Cost: Low**
   375→
   376→Pipeline:
   377→1. Upload image → Next.js Route Handler.
   378→2. **Server-side (Node, via `@xenova/transformers` or Python microservice)**:

   380→   - Run **SAM2 auto-mask** → main subject mask.
   381→   - Run **LaMa (via IOPaint Docker)** on (image, dilated subject mask) → inpainted background.
   382→   - Optionally: bilateral filter the depth, dilate subject mask by ~25 px.
   383→3. Store 4 assets: `original.png`, `depth.png`, `subject_mask.png`, `background_inpainted.png`. Persist URLs in Prisma.
   384→4. **Client-side (R3F shader)**: Variant C shader above + GSAP breathing + SVG turbulence overlay + chromatic aberration. Mouse-driven parallax clamped to ±0.05 UV.

Total compute: ~2–4s/image on a CPU-only Vercel-style box, <1s on GPU.
   387→
   388→### 🥈 Rank 2 — Pure browser-side (Tiefling-style)
   389→**Quality: 7/10 | Feasibility: 9/10 | Effort: Low | Cost: Free (no server GPU)**
   390→
   391→Pipeline:
   392→1. Upload image → stays in browser.
   393→2. `onnxruntime-web` (WebGPU) runs **Depth Anything V2 Small q4f16** (~18MB) → depth map. ~3–5s on M1 Mac, ~20s on mid hardware.
   394→3. `@imgly/background-removal` (WASM) → subject mask + alpha. ~2–3s.
   395→4. Skip LaMa (no good browser inpainting yet); instead use **edge dilation** on the depth + clamp mouse range hard.
   396→5. R3F shader (Variant A) — accept minor stretch artifacts in exchange for zero infra.
   397→
   398→**Why #2**: Zero server compute cost, fully private (image never leaves browser). Great for a free tier / demo. Quality ceiling lower due to no real inpainting. Tiefling's open-source MIT code is a direct reference implementation.
   399→
   400→### 🥉 Rank 3 — Cloud API (Immersity AI or Replicate)
   401→**Quality: 10/10 | Feasibility: 9/10 | Effort: Very Low | Cost: $$**
   402→
   403→Pipeline:
   404→1. Upload → POST to Immersity AI API (or `pollinations/3d-photo-inpainting` on Replicate).
   405→2. Receive: depth map + layered 3D data, OR pre-rendered video.
   406→3. If interactive: render with their viewer / our shader using their layered output. If video: just `<video autoplay loop muted>`.
   407→
   408→**Why #3**: Best quality bar none, least engineering. But: per-image cost ($0.05–0.50), privacy concerns (user images uploaded to third party), rate limits, and vendor lock-in. Good as a **premium upsell** ("Pro" mode) layered on top of Rank 1.

   411→**Quality: 9/10 | Feasibility: 6/10 | Effort: Medium-High | Cost: Medium**

   415→### Rank 5 — Hard 3-layer CSS-only parallax (no depth model)
   416→**Quality: 5/10 | Feasibility: 10/10 | Effort: Very Low | Cost: Free**
   417→
   418→Skip depth model entirely. User (or simple auto-segmentation) provides 3 layers as separate PNGs: sky / midground / subject. Animate with GSAP Variant (d). Cheapest, fastest, but quality limited and requires either manual prep or Rembg + manual layering.

   587→### PART A — Depth Slicing & Multi-Layer Decomposition (6–12 layers)
   588→
   589→We currently get a depth map (`depthUrl`) and an inpainted background (`backgroundUrl`) from the `/api/separate` route, then the VLM nominates 4 layers by name. To go to 6–12 layers we need to **slice the depth map algorithmically** and emit a mask + inpainted plate per band.
   590→
   591→#### A.1 — Depth slicing / quantization (3 concrete algorithms)
   592→
   593→The depth map from image-edit is a continuous grayscale PNG. Normalise to [0,1] then split into N bands. Three algorithms, ranked by quality:
   594→
   595→**(a) Equal-interval slicing** (baseline, naive):
   596→```
   597→function sliceEqual(depthNorm, N) {
   598→  // depthNorm: Float32Array in [0,1]
   599→  // returns: masks: boolean[N][H*W]
   600→  const masks = Array.from({length:N}, () => new Uint8Array(depthNorm.length))
   601→  for (let i=0; i<depthNorm.length; i++) {
   602→    const band = Math.min(N-1, Math.floor(depthNorm[i] * N))
   603→    masks[band][i] = 1
   604→  }
   605→  return masks
   606→}
   607→```
   608→- ✅ Trivial, O(N).
   609→- ❌ Over-segments smooth gradients, under-segments clustered scenes (a portrait with a flat sky wastes 8 bands on sky).
   610→
   611→**(b) K-means 1D on depth histogram** (recommended for adaptive bands):
   612→```
   613→function sliceKMeans1D(depthNorm, K, iters=20) {
   614→  // 1. Build 256-bin histogram of depth values
   615→  const hist = new Float32Array(256)
   616→  for (const v of depthNorm) hist[Math.min(255, Math.floor(v*255))]++
   617→  // 2. Init centroids at equal-quantile positions
   618→  let centroids = quantileInit(hist, K)
   619→  // 3. Lloyd iterations
   620→  for (let it=0; it<iters; it++) {
   621→    const sums = new Float64Array(K), counts = new Float64Array(K)
   622→    for (let b=0; b<256; b++) {
   623→      let nearest = 0, best = Infinity
   624→      for (let k=0; k<K; k++) {
   625→        const d = Math.abs(b - centroids[k])
   626→        if (d < best) { best = d; nearest = k }
   627→      }
   628→      sums[nearest] += b * hist[b]
   629→      counts[nearest] += hist[b]
   630→    }
   631→    for (let k=0; k<K; k++) if (counts[k] > 0) centroids[k] = sums[k] / counts[k]
   632→  }
   633→  // 4. Thresholds = midpoints between sorted centroids
   634→  centroids.sort((a,b)=>a-b)
   635→  const thresholds = []
   636→  for (let k=0; k<K-1; k++) thresholds.push((centroids[k]+centroids[k+1])/2 / 255)
   637→  return sliceByThresholds(depthNorm, thresholds)
   638→}
   639→```
   640→- ✅ Adapts to actual depth distribution. Better for portraits (sky collapses to 1 band, subject gets 5).
   641→- ✅ O(256·K·iters + N) — extremely fast.
   642→- This is the algorithm `sklearn.cluster.KMeans(n_clusters=K, n_init=4)` would produce on the 1D depth histogram.
   643→- Reference: Kaggle "K-Means Clustering - Image Quantization", ML Mastery "K-Means in OpenCV".
   644→
   645→**(c) Multi-Otsu thresholding** (best quality, recommended default):
   646→From scikit-image `threshold_multiotsu` docs (Liao, Chen, Chung 2001):
   647→```
   648→function multiOtsu(depth8, classes=6) {
   649→  // depth8: Uint8Array 0..255
   650→  const hist = new Float64Array(256)
   651→  for (const v of depth8) hist[v]++
   652→  const total = depth8.length
   653→  for (let i=0;i<256;i++) hist[i] /= total
   654→  // Pre-compute cumulative sums P[i] and cumulative means S[i]
   655→  const P = new Float64Array(256), S = new Float64Array(256)
   656→  P[0]=hist[0]; S[0]=0
   657→  for (let i=1;i<256;i++){P[i]=P[i-1]+hist[i]; S[i]=S[i-1]+i*hist[i]}
   658→  // Maximise between-class variance over all (classes-1) threshold combinations
   659→  // For classes<=6 this is O(256^(classes-1)) — ~10^13 for 6 — too slow naively.
   660→  // Use the Liao et al. fast lookup (precompute H(i,j)=sigma between i,j) → O(256^2) preprocessing then O(256^(k-1)) search
   661→  // Pragmatic: for classes<=4 brute force is fine; for 5-10 use the Liao fast form OR fall back to K-means 1D.
   662→  // For our app, classes=4–6: brute force 4-tuple thresholds, with the Liao speedup for >4.
   663→  ...
   664→  return thresholds // array of `classes-1` threshold values 0..255
   665→}
   666→```
   667→- ✅ Maximises between-class variance → statistically optimal band boundaries.
   668→- ✅ Default in scikit-image; battle-tested.
   669→- ⚠️ For >6 classes the brute-force search explodes; use the Liao et al. fast algorithm (precompute `H(i,j)` between-class variance lookup) or fall back to K-means 1D for >6.
   670→- Reference: `skimage.filters.threshold_multiotsu` (linked above), Liao et al. 2001 "A Fast Algorithm for Multilevel Thresholding".
   671→
   672→**Recommendation for our app**: Use **multi-Otsu for 4–6 layers** (the sweet spot — fast + optimal). Switch to **K-means 1D for 7–12 layers** (O(N) regardless of K). Run server-side in `/api/separate` after image-edit returns the depth map; return an array of mask PNGs (one per layer) alongside the existing `backgroundUrl`.

   674→#### A.2 — Layer dilation / edge expansion (no-gap parallax)
   675→
   676→When layers translate independently (mouse parallax), the boundary between depth band `k` and band `k+1` produces a transparent gap. Fix: **dilate each layer's mask** so it overlaps its neighbour by a few px.
   677→
   678→**Sharp (Node) approach** — fastest, already in our project:
   679→```ts
   680→import sharp from 'sharp'
   681→
   682→async function dilateMask(maskPng: Buffer, radiusPx: number): Promise<Buffer> {
   683→  // morphological dilation via repeated max-filter (3x3 box) `radiusPx` times.
   684→  // sharp doesn't expose morphology directly; emulate with threshold-blur-threshold:
   685→  const { data, info } = await sharp(maskPng).raw().toBuffer({resolveWithObject:true})
   686→  // Cheap trick: blur then threshold — produces a "fat" mask.
   687→  const blurred = await sharp(data, {raw:info})
   688→    .blur(radiusPx/2 | 0 || 1)
   689→    .raw().toBuffer()
   690→  // Threshold at 128: any pixel >128 becomes fully opaque
   691→  for (let i=0;i<blurred.length;i+=info.channels){
   692→    const v = blurred[i]
   693→    const out = v > 64 ? 255 : 0
   694→    blurred[i]=blurred[i+1]=blurred[i+2]=out
   695→  }
   696→  return sharp(blurred, {raw:info}).png().toBuffer()
   697→}
   698→```
   699→
   700→**True morphological dilation** (use `jimp` or `@napi-rs/canvas`):
   701→```ts
   702→// Implement classical dilation: out[p] = max(in[q] for q in 3x3 neighbourhood)
   703→// Repeat `radiusPx` times. Equivalent to cv2.dilate(mask, kernel=cv2.ones(3,3), iterations=r)
   704→function dilate(mask: Uint8Array, w: number, h: number, radius: number): Uint8Array {
   705→  let cur = mask.slice()
   706→  for (let it=0; it<radius; it++) {
   707→    const next = new Uint8Array(cur.length)
   708→    for (let y=0;y<h;y++) for (let x=0;x<w;x++){
   709→      let m=0
   710→      for (let dy=-1;dy<=1;dy++) for (let dx=-1;dx<=1;dx++){
   711→        const nx=x+dx, ny=y+dy
   712→        if (nx<0||ny<0||nx>=w||ny>=h) continue
   713→        m = Math.max(m, cur[ny*w+nx])
   714→      }
   715→      next[y*w+x] = m
   716→    }
   717→    cur = next
   718→  }
   719→  return cur
   720→}
   721→```
   722→- **Recommended radius**: 10–30 px (Tiefling uses `expandDepthmapRadius≈25`; Facebook 3D Photos use similar). Scale with target resolution: `radius = Math.round(maxDim * 0.02)`.
   723→- Always dilate **after** slicing, **before** inpainting (so the inpainted plate covers the dilated band).
   724→- Reference: PyImageSearch "OpenCV Morphological Operations", Tiefling `expandDepthmapRadius` (cited in Task 1).

   726→#### A.3 — Per-layer background inpainting
   727→
   728→Three strategies, ranked by quality/effort for our sandbox (VLM + image-edit only, no LaMa/SD):
   729→
   730→**(a) Reuse the single inpainted back plate** (cheapest, already implemented):
   731→- We already have `backgroundUrl` (subject removed). Use it as the universal back plate. Each layer above shows through where its (dilated) mask is empty.
   732→- ✅ Zero extra compute. Works fine for 4–6 layers.
   733→- ❌ For 8–12 layers, mid-depth bands need their *own* occluded content (e.g. a tree in front of mountains in front of sky — removing the tree shouldn't show the sky directly).
   734→
   735→**(b) "Fill from neighboring depth band" trick** (no ML, fast):
   736→- For each layer `k`, the occluded pixels are filled by **stretching the next-farther layer's color** into the gap. Like "content-aware extend" via `sharp.resize` with `fit:'fill'` + edge mirroring.
   737→- ✅ No ML. Fast. Looks plausible for natural scenes.
   738→- ❌ Fails on textured regions (repeats patterns visibly).

   740→**(c) Per-layer VLM re-prompt to image-edit** (the sandbox-native approach):
   741→- For each depth band `k` with name `layerName` (from VLM analysis), re-prompt z-ai image-edit: *"Show what's behind the {layerName} in this image. Same lighting, same style."*
   742→- ✅ Uses tools we already have. Each layer gets its own occluded-content plate.
   743→- ⚠️ 6–12 image-edit calls per project (~3–4s each = 20–50s total). Mitigate with `Promise.all` parallelism + caching.

   747→- Already noted in Task 1: `POST http://iopaint:8000/inpaint {image, mask}` → PNG. ~1s/image on CPU. Use this if we move off the sandbox.

   749→#### A.4 — Order-independent compositing (painter's algorithm with z-sort)
   750→
   751→From GeeksforGeeks "Painter's Algorithm":
   752→> "Sort polygons by depth (decreasing z = farthest first). Paint each in order, nearer polygons overwrite farther ones."
   753→
   754→For our flat depth-banded layers (each a textured plane), this is trivial:
   755→```ts
   756→// sort back-to-front (farthest depth = smallest depth value, where depth=0 is sky, depth=1 is foreground)
   757→const sortedLayers = [...layers].sort((a,b) => a.depth - b.depth)
   758→// render in order; later renders cover earlier where masks overlap
   759→for (const layer of sortedLayers) ctx.drawImage(layer.canvas, 0, 0)
   760→```
   761→- For non-intersecting planes this is **always correct** (no need for BSP/z-buffer).
   762→- For our scene (depth-banded layers with dilated masks) planes never intersect → painter's algorithm is sufficient.
   763→- If we ever add rotated/skewed layers, fall back to per-pixel z-test in the WebGL shader (already supported by depth buffer).
   764→
   765→#### A.5 — Adaptive meshing (vertex-displaced mesh per depth band)
   766→
   767→Currently `AliveWebGL.tsx` uses a single fullscreen quad with UV-offset. Upgrading to **vertex-displaced mesh** gives true parallax geometry and avoids rubber-band stretch.
   768→
   769→**Codrops "Dynamic Terrain Deformation with R3F" pattern** (read in full):
   770→```tsx
   771→<mesh rotation={[-Math.PI/2, 0, 0]}>
   772→  <planeGeometry args={[W, H, GRID_RESOLUTION, GRID_RESOLUTION]} />
   773→  <meshStandardMaterial
   774→    map={colorMap}
   775→    displacementMap={depthMap}
   776→    displacementScale={2}
   777→  />
   778→</mesh>
   779→```
   780→- `GRID_RESOLUTION` = 128 or 256 subdivisions per axis (gives 65K–65K vertices — fine for desktop GPUs).
   781→- Per-vertex Z comes directly from the displacementMap; Three.js handles the GPU upload.
   782→
   783→**For multi-layer (6–12 layers)** — generate one mesh per depth band:
   784→```tsx
   785→sortedLayers.map(layer => (
   786→  <mesh key={layer.id} position={[0, 0, layer.depth * DEPTH_SCALE]}>
   787→    <planeGeometry args={[W, H, GRID, GRID]} />
   788→    <meshBasicMaterial
   789→      map={layer.texture}
   790→      alphaMap={layer.maskTexture}  // the dilated band mask
   791→      transparent
   792→      depthWrite={true}
   793→    />
   794→    {/* Per-vertex Z from band-local depth (cropped from full depth map) */}
   795→    <displacementMap ... />
   796→  </mesh>
   797→))
   798→```
   799→- Use `THREE.PlaneGeometry` (not BufferGeometry) — `displacementMap` works out of the box.
   800→- For 8 layers × 128² verts = 130K verts total — modern GPUs handle this easily at 60fps.
   801→
   802→**Adaptive meshing (Tiefling technique)** — subdivide more at depth discontinuities:
   803→- Compute Sobel edge magnitude of the depth map; where `sobel > threshold`, add 2–4× more subdivisions.
   804→- Implementation: build a custom BufferGeometry where vertex density is high at edges, low in flat regions. ~5K verts total for typical scene (vs 65K uniform).
   805→- ✅ 10× fewer verts, same visual quality.
   806→- ❌ More complex; defer to v3.

   945→**Vanilla alternative** (no library): implement hit-testing + 8 corner handles manually. Konva's reference algorithm for rotated-bbox hit testing:
   946→```ts
   947→const degToRad = (a) => (a/180)*Math.PI
   948→const getCorner = (px, py, dx, dy, angle) => {
   949→  const dist = Math.sqrt(dx*dx + dy*dy)
   950→  angle += Math.atan2(dy, dx)
   951→  return {x: px + dist*Math.cos(angle), y: py + dist*Math.sin(angle)}
   952→}
   953→const getClientRect = (el) => {
   954→  const {x, y, width, height, rotation=0} = el
   955→  const rad = degToRad(rotation)
   956→  const corners = [
   957→    getCorner(x,y, 0,0, rad),
   958→    getCorner(x,y, width,0, rad),
   959→    getCorner(x,y, width,height, rad),
   960→    getCorner(x,y, 0,height, rad),
   961→  ]
   962→  const xs = corners.map(c=>c.x), ys = corners.map(c=>c.y)
   963→  return {x:Math.min(...xs), y:Math.min(...ys), width:Math.max(...xs)-Math.min(...xs), height:Math.max(...ys)-Math.min(...ys)}
   964→}
   965→// Click hit-test:
   966→const isHit = (mousePos, layer) => {
   967→  // inverse-rotate mouse into layer local space, then AABB test
   968→  const rad = degToRad(-layer.rotation)
   969→  const dx = mousePos.x - (layer.x + layer.width/2)
   970→  const dy = mousePos.y - (layer.y + layer.height/2)
   971→  const lx = dx*Math.cos(rad) - dy*Math.sin(rad) + layer.width/2
   972→  const ly = dx*Math.sin(rad) + dy*Math.cos(rad) + layer.height/2
   973→  return lx >= 0 && lx <= layer.width && ly >= 0 && ly <= layer.height
   974→}
   975→```

  1053→#### C.2 — Jitter / boil (Spider-Verse "boiling" effect)
  1054→
  1055→From Camillo Visini's tutorial (read in full — the canonical web implementation):
  1056→```html
  1057→<svg width="0" height="0" style="position:absolute">
  1058→  <filter id="boil">
  1059→    <feTurbulence type="turbulence" baseFrequency="0.012" numOctaves="2" seed="1" result="noise"/>
  1060→    <feDisplacementMap in="SourceGraphic" in2="noise" scale="4"
  1061→      xChannelSelector="R" yChannelSelector="G"/>
  1062→  </filter>
  1063→</svg>
  1064→<img src="layer.png" style="filter:url(#boil)" id="boilTarget"/>
  1065→```
  1066→```ts
  1067→// Animate by cycling baseFrequency every 100ms with small offsets (the "boil" loop)
  1068→const offsets = [-0.002, 0.0014, -0.0011, 0.0021, -0.0017]
  1069→let i = 0
  1070→const baseFreq = 0.012
  1071→const scale = 0.6  // 0..1, "animation scale" — Visini's term
  1072→setInterval(() => {
  1073→  const offset = offsets[i % offsets.length] * scale
  1074→  const turb = document.querySelector('#boil feTurbulence')
  1075→  turb.setAttribute('baseFrequency', (baseFreq + offset).toFixed(5))
  1076→  i++
  1077→}, 100)
  1078→```
  1079→**Key insight from Visini**: don't animate `seed` (causes hard jumps); animate `baseFrequency` by tiny offsets (±0.002) every 100ms. This produces the "hand-drawn wobble" loop. Use prime period (e.g. 4 or 5 offsets) so it never visually repeats.
  1080→- **Per-layer**: each layer gets its own `<filter id="boil-{layerId}">` with different `baseFrequency` and offset array → layers boil out of sync.
  1081→- Recommended `scale` values: 0.3 (subtle), 0.8 (visible), 1.5 (extreme, "Into the Spider-Verse" look).

  1083→#### C.3 — Wave distortion (horizontal sinusoidal displacement)
  1084→
  1085→Different from liquid (which is 2D turbulence). Wave is 1D horizontal sine — like a flag rippling.
  1086→
  1087→```glsl
  1088→// In our existing AliveWebGL fragment shader, add:
  1089→uniform float uWaveAmp;     // ~0.005 UV
  1090→uniform float uWaveFreq;    // ~12.0 cycles across image
  1091→uniform float uWaveTime;
  1092→varying vec2 vUv;
  1093→
  1094→void main() {
  1095→  float wave = sin(vUv.x * uWaveFreq + uWaveTime * 1.6) * uWaveAmp;
  1096→  vec2 uv = vUv + vec2(0.0, wave);  // vertical displacement from horizontal sine
  1097→  gl_FragColor = texture2D(uImage, uv);
  1098→}
  1099→```

  1222→#### C.9 — Mouse velocity influence (layers react to mouse SPEED not just position)
  1223→
  1224→Currently parallax follows `mousePos`. Velocity adds inertia: when mouse moves fast, layers "lag" and overshoot.
  1225→
  1226→```ts
  1227→const mouse = {x:0, y:0, px:0, py:0, vx:0, vy:0}
  1228→window.addEventListener('pointermove', (e) => {
  1229→  mouse.x = e.clientX / window.innerWidth - 0.5
  1230→  mouse.y = e.clientY / window.innerHeight - 0.5
  1231→})
  1232→// In rAF loop:
  1233→function tick() {
  1234→  // velocity = (current - previous) / dt
  1235→  mouse.vx = (mouse.x - mouse.px) * 60  // assume 60fps
  1236→  mouse.vy = (mouse.y - mouse.py) * 60
  1237→  mouse.px = mouse.x
  1238→  mouse.py = mouse.y
  1239→  // Low-pass filter (smoothing)
  1240→  const smoothVx = lerp(state.svx, mouse.vx, 0.08)
  1241→  const smoothVy = lerp(state.svy, mouse.vy, 0.08)
  1242→  // Apply: position = mousePos * depth + velocity * depth * inertia
  1243→  layers.forEach(layer => {
  1244→    const tx = mouse.x * layer.depth * 40 + smoothVx * layer.depth * 80
  1245→    const ty = mouse.y * layer.depth * 40 + smoothVy * layer.depth * 80
  1246→    gsap.set(layer.el, {x:tx, y:ty})
  1247→  })
  1248→  requestAnimationFrame(tick)
  1249→}
  1250→```
  1251→- Velocity amplitude (80) is 2× position amplitude (40) → fast moves dominate.
  1252→- Low-pass filter (0.08) prevents jitter on slow moves.

  1254→#### C.10 — Spring physics per layer (independent stiffness/damping/mass)
  1255→
  1256→From Maxime Heckel's article (read in full — the canonical explanation). Spring physics math:
  1257→```
  1258→Fspring = -k * (x - restLength)        // Hooke's law
  1259→Fdamping = -d * v                      // damping force
  1260→a = (Fspring + Fdamping) / m           // Newton's 2nd law
  1261→v += a * dt                            // integrate velocity
  1262→x += v * dt                            // integrate position
  1263→```
  1264→JavaScript implementation (one layer):
  1265→```ts
  1266→class Spring {
  1267→  constructor({stiffness=100, damping=10, mass=1, rest=0} = {}) {
  1268→    this.k = stiffness; this.d = damping; this.m = mass
  1269→    this.rest = rest; this.x = rest; this.v = 0
  1270→  }
  1271→  setTarget(t) { this.rest = t }
  1272→  tick(dt = 1/60) {
  1273→    const Fspring = -this.k * (this.x - this.rest)
  1274→    const Fdamping = -this.d * this.v
  1275→    const a = (Fspring + Fdamping) / this.m
  1276→    this.v += a * dt
  1277→    this.x += this.v * dt
  1278→    return this.x
  1279→  }
  1280→}
  1281→```
  1282→- **Per-layer config**: add `springStiffness`, `springDamping`, `springMass` to `LayerAnimationConfig`. Default (Framer's defaults): stiffness=100, damping=10, mass=1.
  1283→- **Tuning guide**: lower stiffness = looser/slower; higher damping = less overshoot; higher mass = sluggish.
  1284→- **Use Framer Motion's `useSpring`** for declarative version:
  1285→```tsx
  1286→const x = useSpring(0, {stiffness: layer.springStiffness, damping: layer.springDamping, mass: layer.springMass})
  1287→useEffect(() => x.set(mouseX * depth * 40), [mouseX])
  1288→return <motion.div style={{x}} />
  1289→```

  1291→#### C.11 — Phase offset between layers (staggered animation start so layers never sync)
  1292→
  1293→From motion.dev `stagger()` docs (read in full):
  1294→```ts
  1295→import { animate, stagger } from 'motion'
  1296→
  1297→// Stagger all layers' entrance by 80ms each
  1298→animate('.alive-layer', {opacity: [0, 1]}, {delay: stagger(0.08)})
  1299→
  1300→// Stagger from center outward
  1301→animate('.alive-layer', {scale: [0.9, 1]}, {delay: stagger(0.05, {from: 'center'})})
  1302→
  1303→// With easing for non-linear stagger
  1304→animate('.alive-layer', {y: [20, 0]}, {delay: stagger(0.1, {ease: 'easeOut'})})
  1305→
  1306→// Negative startDelay = layers start mid-animation (never sync)
  1307→animate('.alive-layer', {rotate: [0, 360]}, {
  1308→  duration: 20, repeat: Infinity, ease: 'linear',
  1309→  delay: stagger(0.7, {startDelay: -2})  // each layer starts 0.7s into the loop, offset
  1310→})
  1311→```
  1312→- `stagger()` options: `startDelay`, `from` ("first"|"center"|"last"|index), `ease`.
  1313→- **For continuous loops** (breathing, sway), the cleanest pattern: each layer's CSS animation has a `animation-delay` of `-1.3s * layerDepth` (negative = starts mid-cycle) → layers never visually sync. Use prime offsets (1.3, 2.7, 4.1…) to maximise desync.
  1314→- Reference: CSS-Tricks "Different Approaches for Creating a Staggered Animation".

  1316→#### C.12 — Simplex noise per-layer motion (curl noise displacement)
  1317→
  1318→```bash
  1319→npm install simplex-noise@4
  1320→```
  1321→```ts
  1322→import { createNoise2D } from 'simplex-noise'
  1323→
  1324→const noise2D = createNoise2D(() => Math.random())
  1325→// Per-layer noise field seeded differently:
  1326→const layerNoise = createNoise2D(() => layer.seed * 1337)
  1327→
  1328→function tick(dt: number, t: number) {
  1329→  for (const layer of layers) {
  1330→    // Sample 2 noise points slightly apart → "curl" gives divergence-free field
  1331→    const eps = 0.001
  1332→    const n1 = layerNoise(t * 0.05,            layer.seed)
  1333→    const n2 = layerNoise(t * 0.05 + eps,      layer.seed)
  1334→    const n3 = layerNoise(layer.seed,          t * 0.05)
  1335→    const n4 = layerNoise(layer.seed + eps,    t * 0.05)
  1336→    const dx = (n2 - n1) / eps * 20  // scale to px
  1337→    const dy = (n4 - n3) / eps * 20
  1338→    layer.el.style.transform = `translate(${dx}px, ${dy}px)`
  1339→  }
  1340→  requestAnimationFrame((nt) => tick(dt, nt / 1000))
  1341→}
  1342→```
  1343→- **Curl noise** (Bridson et al.): gives smooth divergence-free velocity fields → particles/layers flow without clumping.
  1344→- **Per-layer**: different `seed` (or different noise function instance) per layer → independent motion.
  1345→- Amplitude: ~10–30px. Time scale: `0.02–0.1` (slow drift).

  1374→#### D.2 — Snow / rain (canvas particle system)
  1375→
  1376→Lighter than `particles.js` (which is 50KB); a custom canvas impl is ~50 lines:
  1377→```ts
  1378→class Snow {
  1379→  canvas = document.createElement('canvas')
  1380→  ctx = this.canvas.getContext('2d')!
  1381→  particles = Array.from({length: 120}, () => ({
  1382→    x: Math.random()*innerWidth, y: Math.random()*innerHeight,
  1383→    r: 1+Math.random()*3, vy: 0.5+Math.random()*1.5, vx: -0.3+Math.random()*0.6,
  1384→    sway: Math.random()*Math.PI*2
  1385→  }))
  1386→  tick() {
  1387→    const {ctx, canvas, particles} = this
  1388→    ctx.clearRect(0,0,canvas.width,canvas.height)
  1389→    ctx.fillStyle = 'rgba(255,255,255,0.85)'
  1390→    for (const p of particles) {
  1391→      p.sway += 0.02
  1392→      p.x += p.vx + Math.sin(p.sway)*0.5
  1393→      p.y += p.vy
  1394→      if (p.y > canvas.height) p.y = -10, p.x = Math.random()*canvas.width
  1395→      ctx.beginPath()
  1396→      ctx.arc(p.x, p.y, p.r, 0, Math.PI*2)
  1397→      ctx.fill()
  1398→    }
  1399→    requestAnimationFrame(() => this.tick())
  1400→  }
  1401→}
  1402→```
  1403→- For rain: larger `vy` (8–15), elongated streaks (use `ctx.fillRect` with 1×8px), blue-grey tint.
  1404→- For snow: slow `vy` (0.5–1.5), circular, white, with horizontal sway.

  1406→#### D.3 — Light rays / god rays (radial blur post-process)
  1407→
  1408→From Andrew Berg's article (Kenny Mitchell, GPU Gems 3 — read in full):
  1409→
  1410→**The technique** (3 steps):
  1411→1. Render the scene with light source = white, occluders = black → "occlusion texture".
  1412→2. Apply radial blur toward the light source in a fragment shader:
  1413→```glsl
  1414→// Volumetric light scattering shader (Mitchell, GPU Gems 3)
  1415→uniform sampler2D uOcclusion;
  1416→uniform vec2 uLightPos;       // screen-space light position
  1417→uniform float uDecay;         // 0.95
  1418→uniform float uDensity;       // 0.5
  1419→uniform float uWeight;        // 0.4
  1420→uniform float uExposure;      // 0.2
  1421→uniform int uSamples;         // 80–100
  1422→
  1423→void main() {
  1424→  vec2 uv = vUv;
  1425→  vec2 dir = uv - uLightPos;
  1426→  dir *= uDensity;  // step size
  1427→  float illum = 1.0;
  1428→  vec4 col = texture2D(uOcclusion, uv);
  1429→  for (int i=0; i<100; i++) {
  1430→    if (i >= uSamples) break;
  1431→    uv -= dir;
  1432→    vec4 c = texture2D(uOcclusion, uv);
  1433→    c *= illum * uWeight;
  1434→    col += c;
  1435→    illum *= uDecay;
  1436→  }
  1437→  gl_FragColor = col * uExposure;
  1438→}
  1439→```
  1440→3. Additively blend the result over the original scene render.
  1441→
  1442→**Cheaper pseudo-volumetric** (three.jsdemos.com pattern — for our app):
  1443→- Use a `ConeGeometry` mesh, `MeshBasicMaterial` with `transparent:true, opacity:0.15, blending:AdditiveBlending, side:DoubleSide, depthWrite:false`.
  1444→- Position the cone tip at the light source, expand toward the scene.
  1445→- Animate opacity via a sine wave (slow pulse).
  1446→

  1478→#### D.5 — Depth-aware dust motes (parallax particles)
  1479→
  1480→Upgrade the existing `Particles.tsx` to be depth-aware:
  1481→```ts
  1482→// Each dust particle gets a depth value [0..1]
  1483→const motes = Array.from({length: 40}, () => ({
  1484→  x: Math.random(), y: Math.random(),
  1485→  z: 0.1 + Math.random()*0.9,    // depth!
  1486→  r: 0.5 + Math.random()*1.5,
  1487→  drift: Math.random()*Math.PI*2
  1488→}))
  1489→
  1490→function tick(t, mouseX, mouseY) {
  1491→  for (const m of motes) {
  1492→    // close motes (high z) move MORE with mouse
  1493→    const px = m.x + Math.sin(t*0.0003 + m.drift) * 0.04
  1494→    const py = m.y - (t*0.00005 * (0.5 + m.z)) % 1
  1495→    const screenX = (px * canvas.width) + mouseX * m.z * 60
  1496→    const screenY = (py * canvas.height) + mouseY * m.z * 60
  1497→    // close motes are larger and brighter
  1498→    const size = m.r * (0.5 + m.z * 0.8)
  1499→    const alpha = 0.2 + m.z * 0.5
  1500→    ctx.fillStyle = `rgba(255,240,200,${alpha})`
  1501→    ctx.beginPath(); ctx.arc(screenX, screenY, size, 0, Math.PI*2); ctx.fill()
  1502→  }
  1503→}
  1504→```
  1505→- Each particle samples its `z` from the depth map at its position → true depth-aware parallax.

  1507→#### D.6 — Volumetric light (pseudo-volumetric shafts)

  1509→The "god rays" radial-blur technique (D.3) is the canonical web fake. True volumetric requires ray-marching which is too heavy for browser.

  1574→#### E.2 — CSS `@keyframes` vs Web Animations API
  1575→
  1576→From CSS-Tricks "Comparison of Animation Technologies" (read in full) + MDN "CSS and JavaScript animation performance":
  1577→
  1578→| Use case | CSS `@keyframes` | Web Animations API (WAAPI) | Motion / GSAP |
  1579→|---|---|---|---|
  1580→| Simple infinite loops (breathing, sway, hue drift) | ✅ Best — declarative, GPU-accelerated, pauses cleanly | OK | OK |
  1581→| Imperative / scrubbable / one-shot | ❌ | ✅ Best — `animate()` returns Animation object, can `pause()`/`seek`/`reverse` | ✅ |
  1582→| Spring physics | ❌ | ❌ (no native spring) | ✅ (Motion `useSpring`, GSAP `CustomEase`) |
  1583→| Sequencing (timeline) | ❌ Painful with `animation-delay` | OK with `Promise.all` of `.finished` | ✅ Best (GSAP `timeline()`, Motion `animate([segments])`) |
  1584→| Stagger | ❌ Manual `nth-child` | OK via JS | ✅ Motion `stagger()`, GSAP `stagger` config |
  1585→| Performance | ✅ Best — compositor thread | ✅ Same engine as CSS, same perf | ✅ Same |
  1586→
  1587→**Recommendation for our app**:
  1588→- **CSS `@keyframes` + `@property`** for all the infinite subtle loops (breathing, sway, twist, glow, hue drift, focus pull, shadow drift, boil). These run on the compositor thread, don't need JS, and pause cleanly on tab-inactive.
  1589→- **Motion's `animate()`** (which wraps WAAPI) for: entrance/exit, mouse-driven parallax (imperative), timeline playback, layer reordering transitions.
  1590→- **GSAP** (already in our project per Task 1) for: complex orchestrated reveals, scroll-triggered sequences. Skip if not needed — Motion covers most cases.
  1591→
  1592→#### E.3 — Animation orchestration (sequencing / delays / staggered starts)
  1593→
  1594→**Motion.dev pattern** (from motion.dev/docs/stagger — read in full):
  1595→```ts
  1596→import { animate, stagger } from 'motion'
  1597→
  1598→// Sequential segments
  1599→animate([
  1600→  ['.alive-layer', {opacity: [0, 1]}, {delay: stagger(0.08)}],
  1601→  ['.vignette',    {opacity: [0, 1]}, {delay: 0.3}],
  1602→  ['.particles',   {scale: [0, 1]},   {delay: 0.5}]
  1603→])
  1604→// Each segment waits for the previous to complete (or use `at:` for overlap)
  1605→```
  1606→
  1607→**GSAP timeline pattern**:
  1608→```ts
  1609→const tl = gsap.timeline()
  1610→tl.from('.alive-layer', {opacity:0, stagger:0.08, duration:0.6})
  1611→  .from('.vignette',    {opacity:0, duration:0.4}, '-=0.3')  // overlap 0.3s
  1612→  .from('.particles',   {scale:0, duration:0.5}, '<')         // start same time as prev
  1613→```
  1614→
  1615→**For continuous "alive" loops**: don't use a timeline. Use per-layer CSS animations with prime-duration periods and negative `animation-delay` offsets so layers never sync:
  1616→```css
  1617→.alive-layer:nth-child(1) { animation: breath 6.2s -1.3s infinite alternate; }
  1618→.alive-layer:nth-child(2) { animation: breath 6.2s -2.7s infinite alternate; }
  1619→.alive-layer:nth-child(3) { animation: breath 6.2s -4.1s infinite alternate; }
  1620→.alive-layer:nth-child(4) { animation: breath 6.2s -5.3s infinite alternate; }
  1621→/* negative delay = layer starts mid-cycle → never visually synchronised */
  1622→```
