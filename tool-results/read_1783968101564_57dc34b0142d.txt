    17‚ÜíStage Summary:
    18‚Üí- **Top recommendation**: Hybrid pipeline ‚Äî run **Depth Anything V2 (Small or Base)** server-side via Python (or HuggingFace Inference API / Replicate) to get a depth map, then run **SAM2 auto-mask** to identify the main subject, then **LaMa (via IOPaint)** to inpaint the background behind the subject. Pure UV-offset creates stretched "rubber band" artifacts at depth discontinuities; this is exactly what Immersity/LeiaPix/3D-Photo-Inpainting solve by hallucinating the background behind foreground objects. We MUST replicate this or the effect looks cheap.
    20‚Üí- **Critical insight #2**: Limit camera/mouse motion to a small range (~¬±10¬∞ rotation or a few tens of px translation). Google's Cinematic Photos team confirmed this is the key to hiding artifacts without expensive per-frame inpainting.

    51‚Üí### 2. Layer Separation from Depth Maps
    52‚Üí
    53‚ÜíGiven a continuous grayscale depth map, splitting into N discrete layers:
    54‚Üí
    55‚Üí**a) Equal-width thresholding (simplest)**: normalize depth to [0,1], slice into N bands. Easy but tends to over-segment smooth gradients.
    56‚Üí
    57‚Üí**b) K-means / quantile clustering** (recommended for 3‚Äì5 layers): cluster depth pixels into K bins using `sklearn.cluster.KMeans` or 1-D k-means on the depth histogram. Adapts to actual scene depth distribution. Reference: "Object-Background Region Separation in Depth Image using k-Means Clustering" (KISM 2021).
    58‚Üí
    59‚Üí**c) Bilateral filtering first** (used by Shih et al. 3D-Photo-Inpainting): edge-preserving smoothing of the depth map before clustering ‚Üí cleaner layer boundaries that align with image edges.
    60‚Üí
    61‚Üí**d) Watershed / connected-component labeling**: produce spatially coherent regions (not just depth bands) ‚Üí useful for "subject vs background" rather than "near vs far".
    62‚Üí
    63‚Üí**e) Adaptive layering (Layered Depth Image, LDI)**: the gold standard from Shih et al. ‚Äî splits along depth *discontinuities* with explicit pixel connectivity, producing a graph of front-to-back layers each with its own inpainted color+depth. This is what 3D Photo Inpainting / Immersity use.
    64‚Üí
    65‚ÜíFor our app, **b + c (bilateral-filter then 1-D k-means with K=3‚Äì5)** is the practical sweet spot for "foreground / midground / background / sky" decomposition. Save the cluster centroids as depth thresholds and emit per-layer masks as PNGs.
    66‚Üí
    67‚Üí### 3. Segmentation-Based Separation (SAM / SAM2)
    68‚Üí
    69‚Üí**Meta SAM2** (`facebookresearch/sam2`, Apache 2.0) is the SOTA. Two modes:
    70‚Üí- **Promptable**: click/box/mask ‚Üí precise mask. Great for "user clicks subject".
    71‚Üí- **Automatic mask generator**: produces all plausible masks. Then we filter by area / centrality to pick the main subject.
    72‚Üí
    73‚Üí**Sky detection**: dedicated models exist (SkyFinder dataset, "Segmenting Sky Pixels in Images"), but a simpler heuristic often suffices ‚Äî combine (a) top-of-image region, (b) blue-ish / bright color, (c) **far depth** (depth value in the top decile). Can also use a semantic seg model like **ADE20K-trained OneFormer / Mask2Former** which has a "sky" class out of the box.
    74‚Üí
    75‚Üí**Ground detection**: similar ‚Äî bottom-of-image + dark/uniform color + near depth. Or ADE20K class "floor / grass / road / earth".
    76‚Üí
    77‚Üí**Subject detection**: SAM2 auto-mask ‚Üí pick the largest non-sky/non-ground mask whose centroid is in the central 60% of the frame. Optionally refine edges with **Matting-Anything** or **MODNet** for hair-level fidelity (important for portraits).
    78‚Üí
    79‚Üí**Recommended pipeline**:
    80‚Üí1. Run ADE20K semantic seg (1 forward pass) ‚Üí labels every pixel as sky / ground / person / building / etc.
    81‚Üí2. Run SAM2 auto-mask ‚Üí instance-level subject candidates.
    82‚Üí3. Cross-reference: subject = SAM2 mask overlapping "person" or central non-background ADE20K labels.
    83‚Üí
    84‚ÜíFor a Next.js app, run server-side via the `transformers.js` SAM2 port or a Python microservice. **transformers.js** can run SAM2 in Node.js directly ‚Äî no Python needed.
    85‚Üí
    86‚Üí### 4. Background Removal + Inpainting
    87‚Üí
    88‚Üí**Background removal**:
    89‚Üí- **Rembg** (Python, MIT) ‚Äî wraps U2Net / BRIA RMBG / IS-Net. CLI + library. Production-grade, ~1‚Äì2s per image on CPU. Best free option.
    90‚Üí- **@imgly/background-removal** (JS, runs in browser or Node via WASM) ‚Äî drop-in for Next.js, no Python.
    91‚Üí
    92‚Üí**Inpainting the occluded background** (critical for parallax quality):
    93‚Üí- **LaMa** (Resolution-robust Large Mask Inpainting with Fourier Convolutions, Samsung Research) ‚Äî SOTA for large mask inpainting, generalizes to 2K res despite 256√ó256 training. Fast (~1s/image on GPU). Apache 2.0.
    94‚Üí- **IOPaint** (formerly Lama-Cleaner, MIT) ‚Äî self-hostable web app + Python lib + HTTP API that wraps LaMa, Stable Diffusion inpainting, MAT, ZITS, FcF, Manga, PaintByExample, and several matting models. **This is the swiss-army knife.** Docker container ‚Üí Next.js calls `POST http://iopaint:8000/inpaint` with image + mask.
    95‚Üí- **Stable Diffusion Inpainting** ‚Äî highest quality for complex scenes, but slow (5‚Äì15s on GPU) and may "hallucinate" extra content.
    96‚Üí- **3D Photo Inpainting** (Shih et al., CVPR 2020, `vt-vl-lab/3d-photo-inpainting`) ‚Äî the canonical method. Takes RGB+depth, outputs a Layered Depth Image (LDI) with color+depth for occluded regions. Heavy (~10 min on A100 per image via Replicate). Available as `pollinations/3d-photo-inpainting` on Replicate. **Best quality**, expensive.
    97‚Üí- **SLIDE** (ICCV 2021) ‚Äî soft layering variant; handles hair/foliage better. No official code; community reimplementations exist.
    98‚Üí
    99‚Üí**Practical recommendation for our app**:
   100‚Üí1. Default: Rembg (or `@imgly/background-removal`) for subject alpha matte.
   101‚Üí2. **Dilate the subject mask by ~20‚Äì40 px** (this is what Facebook 3D Photos / Tiefling's `expandDepthmapRadius` do ‚Äî exposes a band of background that can be revealed when subject shifts).
   102‚Üí3. Run **LaMa via IOPaint** on the dilated mask ‚Üí reconstructs background where the subject used to be. Store this as `background_inpainted.png`.
   103‚Üí4. For premium mode: route through `pollinations/3d-photo-inpainting` on Replicate for the full LDI treatment (cached, async).

   149‚Üí### 7. The "LSD Alive" Effect ‚Äî Subtle Animation Toolkit
   150‚Üí
   151‚ÜíThe "alive" feeling is the *combination* of several subtle, low-amplitude animations. Each alone is barely noticeable; together they create the magic. Key techniques (ranked by impact):
   152‚Üí
   153‚Üí1. **Mouse / gyro parallax** (the foundation). Map `mouseX/Y` (or `deviceorientation`) to camera/layer offset, multiplied by per-layer depth. Smooth with lerp (damping ~0.05‚Äì0.1).
   154‚Üí2. **Breathing / pulsing** ‚Äî slow (4‚Äì8s period) `scale: 1.0 ‚Üî 1.015`. Apply to whole image and slightly stronger to foreground.
   155‚Üí3. **Sway** ‚Äî gentle `rotate: ¬±0.3¬∞` oscillation on a 6‚Äì10s sine. Subtle but very "alive".
   156‚Üí4. **Liquid / wavy distortion** ‚Äî `<feTurbulence>` SVG filter animated via `seed` or `baseFrequency`, applied with low `scale` (1‚Äì3px displacement). Gives the "water surface" shimmer. Optional per-region.
   157‚Üí5. **Floating particles** ‚Äî CSS / canvas particles (dust motes, light specks) drifting upward or with mouse. ~10‚Äì30 particles, low opacity.
   158‚Üí6. **Light shimmer** ‚Äî slow gradient overlay sweeping across (radial highlight following mouse, or auto-cycling). ~2‚Äì5% opacity.
   159‚Üí7. **Chromatic aberration / RGB split** ‚Äî shift R/B channels 1‚Äì3px based on mouse distance from center. Increases at edges (radial).
   160‚Üí8. **Depth-of-field shift** ‚Äî blur the far layer (or near layer) alternately on a 8‚Äì12s cycle. `filter: blur(0‚Äì4px)`.
   161‚Üí9. **Ken Burns** ‚Äî slow zoom (1.0 ‚Üí 1.05 over 20‚Äì30s) + sub-pixel pan. Good as a base when there's no mouse input.
   162‚Üí10. **Vignette pulse** ‚Äî very subtle darkening at edges pulsing with breathing.
   163‚Üí
   164‚ÜíAmplitude guidelines: any single effect should be barely perceptible when stared at. Combined, they shouldn't draw attention to themselves. The user should just feel "the image is breathing".

   166‚Üí### 8. Code Examples
   167‚Üí
   168‚Üí**a) Codrops / canonical WebGL fragment shader** (the foundation of LeiaPix/DepthFlow/Tiefling):
   169‚Üí```glsl
   170‚Üí// Fragment shader: original image + depth map ‚Üí parallax with mouse
   171‚Üíuniform sampler2D originalImage;
   172‚Üíuniform sampler2D depthImage;
   173‚Üíuniform vec2 mouse;  // normalized -1..1, damped
   174‚Üíuniform float depthStrength;  // ~0.02 for subtle
   175‚Üívarying vec2 vUv;
   176‚Üí
   177‚Üívoid main() {
   178‚Üí    vec4 depth = texture2D(depthImage, vUv);
   179‚Üí    // White = close ‚Üí moves more with mouse; black = far ‚Üí stays put
   180‚Üí    vec2 offset = mouse * depth.r * depthStrength;
   181‚Üí    gl_FragColor = texture2D(originalImage, vUv + offset);
   182‚Üí}
   183‚Üí```
   184‚ÜíThat's the whole effect. ~6 lines of GLSL. From the Codrops tutorial by Yuri Artiukh.
   185‚Üí
   186‚Üí**b) React Three Fiber implementation** (R3F + drei):
   187‚Üí```tsx
   188‚Üíimport { Canvas, useFrame, useLoader } from '@react-three/fiber'
   189‚Üíimport { TextureLoader } from 'three'
   190‚Üíimport { useRef, useMemo } from 'react'
   191‚Üíimport * as THREE from 'three'
   192‚Üí
   193‚Üíconst vertexShader = `
   194‚Üí  varying vec2 vUv;
   195‚Üí  void main() { vUv = uv; gl_Position = vec4(position, 1.0); }
   196‚Üí`
   197‚Üíconst fragmentShader = `
   198‚Üí  uniform sampler2D uImage; uniform sampler2D uDepth;
   199‚Üí  uniform vec2 uMouse; uniform float uTime;
   200‚Üí  varying vec2 vUv;
   201‚Üí  void main() {
   202‚Üí    vec4 d = texture2D(uDepth, vUv);
   203‚Üí    // subtle breathing + mouse parallax
   204‚Üí    float breath = sin(uTime * 0.8) * 0.003 * d.r;
   205‚Üí    vec2 off = uMouse * d.r * 0.03 + vec2(breath, breath * 0.5);
   206‚Üí    gl_FragColor = texture2D(uImage, vUv + off);
   207‚Üí  }
   208‚Üí`
   209‚Üí
   210‚Üífunction AlivePlane({ img, depth }) {
   211‚Üí  const mat = useRef<THREE.ShaderMaterial>(null!)
   212‚Üí  const tex = useLoader(TextureLoader, img)
   213‚Üí  const depthTex = useLoader(TextureLoader, depth)
   214‚Üí  const mouse = useMemo(() => new THREE.Vector2(0, 0), [])
   215‚Üí  useFrame((state, delta) => {
   216‚Üí    // smooth mouse follow
   217‚Üí    mouse.x += (state.pointer.x - mouse.x) * 0.05
   218‚Üí    mouse.y += (state.pointer.y - mouse.y) * 0.05
   219‚Üí    mat.current.uniforms.uMouse.value = mouse
   220‚Üí    mat.current.uniforms.uTime.value += delta
   221‚Üí  })
   222‚Üí  return (
   223‚Üí    <mesh>
   224‚Üí      <planeGeometry args={[2, 2]} />
   225‚Üí      <shaderMaterial ref={mat} vertexShader={vertexShader} fragmentShader={fragmentShader}
   226‚Üí        uniforms={{
   227‚Üí          uImage:  { value: tex },
   228‚Üí          uDepth:  { value: depthTex },
   229‚Üí          uMouse:  { value: mouse },
   230‚Üí          uTime:   { value: 0 },
   231‚Üí        }} />
   232‚Üí    </mesh>
   233‚Üí  )
   234‚Üí}
   235‚Üí```

   312‚Üí### 9. Depth-Map-to-Displacement Technique ‚Äî Deep Dive (the LeiaPix/Immersity method)
   313‚Üí
   314‚ÜíThis is the single most important technique for our project. Two rendering variants:
   315‚Üí
   316‚Üí**Variant A ‚Äî Fragment-shader UV offset** (Codrops, Depthy, Pixi DisplacementFilter):
   317‚Üí- 1 fullscreen quad, fragment shader samples depth, offsets UV.
   318‚Üí- ‚úÖ Trivial to implement, ~6 lines GLSL, runs at 60fps on any GPU.
   319‚Üí- ‚ùå Produces **"rubber band" / stretch artifacts** at depth discontinuities: when a foreground pixel shifts, it leaves a hole filled by the next pixel's color ‚Üí smeared edges.
   320‚Üí
   321‚Üí**Variant B ‚Äî Vertex-displaced mesh** (Tiefling, three.js forum examples):
   322‚Üí- Subdivided plane geometry; each vertex's Z = depth value.
   323‚Üí- ‚úÖ True parallax geometry, handles occlusion correctly via depth buffer.
   324‚Üí- ‚ùå Mesh resolution tradeoff: vertex per pixel = too many verts (2M for 1080p). Need adaptive meshing.
   325‚Üí
   326‚Üí**Variant C ‚Äî Layered / multi-plane** (Facebook 3D Photo, Immersity, 3D-Photo-Inpainting):
   327‚Üí- N depth planes, each with its own (potentially inpainted) color texture.
   328‚Üí- ‚úÖ Best quality: occlusion solved by inpainting behind each foreground layer.
   329‚Üí- ‚ùå Requires per-layer background inpainting; more complex asset pipeline.
   330‚Üí
   331‚Üí**The key quality unlocks (do ALL of these)**:
   332‚Üí1. **Dilate the depth map at object edges** before using as displacement (Tiefling's `expandDepthmapRadius`). This pre-extends the background so when foreground shifts, there's real background to show.
   333‚Üí2. **Inpaint the background** behind the main subject (LaMa). Ship `background_inpainted.png` as a separate texture sampled where `depth < subject_depth`.
   334‚Üí3. **Limit motion range** (Google Cinematic Photos insight): clamp mouse offset to ¬±10¬∞ or ¬±30‚Äì50 px. Beyond that, artifacts become visible.
   335‚Üí4. **Use easing / damping** on mouse so motion never exceeds comfortable range.
   336‚Üí5. **Apply bilateral filtering** to the depth map to remove noisy speckles before slicing.
   337‚Üí6. For best results, render with **variant B (vertex mesh) using the inpainted background as a second layer behind** ‚Äî gets ~90% of Immersity quality at 10% of the compute.
   338‚Üí
   339‚Üí**Concrete shader with occlusion-aware background fill** (Variant C, simplified):
   340‚Üí```glsl
   341‚Üíuniform sampler2D uForeground;   // original image
   342‚Üíuniform sampler2D uBackground;   // LaMa-inpainted background (subject removed)
   343‚Üíuniform sampler2D uDepth;
   344‚Üíuniform sampler2D uSubjectMask;  // 1=subject, 0=background
   345‚Üíuniform vec2 uMouse;
   346‚Üíuniform float uTime;
   347‚Üívarying vec2 vUv;
   348‚Üí
   349‚Üívoid main() {
   350‚Üí  vec4 d = texture2D(uDepth, vUv);
   351‚Üí  float mask = texture2D(uSubjectMask, vUv).r;
   352‚Üí  float breath = sin(uTime * 0.6) * 0.002;
   353‚Üí
   354‚Üí  // Foreground shifts more (subject), background less
   355‚Üí  vec2 bgOff = uMouse * (1.0 - d.r) * 0.015 + vec2(breath);
   356‚Üí  vec2 fgOff = uMouse * d.r * 0.05 + vec2(breath * 2.0);
   357‚Üí
   358‚Üí  vec3 bg = texture2D(uBackground, vUv + bgOff).rgb;
   359‚Üí  vec3 fg = texture2D(uForeground, vUv + fgOff).rgb;
   360‚Üí
   361‚Üí  // Use the displaced mask to decide which layer shows
   362‚Üí  float displacedMask = texture2D(uSubjectMask, vUv + fgOff).r;
   363‚Üí  vec3 color = mix(bg, fg, displacedMask);
   364‚Üí
   365‚Üí  gl_FragColor = vec4(color, 1.0);
   366‚Üí}
   367‚Üí```

   371‚Üí## RANKED TECHNIQUES (for our Next.js "alive image" app)
   372‚Üí
   373‚Üí### ü•á Rank 1 ‚Äî Hybrid: Depth Anything V2 + LaMa + Three.js shader (RECOMMENDED)
   374‚Üí**Quality: 9/10 | Feasibility: 8/10 | Effort: Medium | Cost: Low**
   375‚Üí
   376‚ÜíPipeline:
   377‚Üí1. Upload image ‚Üí Next.js Route Handler.
   378‚Üí2. **Server-side (Node, via `@xenova/transformers` or Python microservice)**:
   379‚Üí   - Run **Depth Anything V2 Small/Base** ‚Üí depth map PNG.
   380‚Üí   - Run **SAM2 auto-mask** ‚Üí main subject mask.
   381‚Üí   - Run **LaMa (via IOPaint Docker)** on (image, dilated subject mask) ‚Üí inpainted background.
   382‚Üí   - Optionally: bilateral filter the depth, dilate subject mask by ~25 px.
   383‚Üí3. Store 4 assets: `original.png`, `depth.png`, `subject_mask.png`, `background_inpainted.png`. Persist URLs in Prisma.
   384‚Üí4. **Client-side (R3F shader)**: Variant C shader above + GSAP breathing + SVG turbulence overlay + chromatic aberration. Mouse-driven parallax clamped to ¬±0.05 UV.
   385‚Üí
   386‚Üí**Why #1**: Best quality-per-effort. Uses SOTA depth + SOTA inpainting without paying for 3D-Photo-Inpainting's 10-min/A100 cost. The `transformers.js` route means **no Python required** if we use `@xenova/transformers` + `@imgly/background-removal` + IOPaint-as-a-service. Total compute: ~2‚Äì4s/image on a CPU-only Vercel-style box, <1s on GPU.
   387‚Üí
   388‚Üí### ü•à Rank 2 ‚Äî Pure browser-side (Tiefling-style)
   389‚Üí**Quality: 7/10 | Feasibility: 9/10 | Effort: Low | Cost: Free (no server GPU)**
   390‚Üí
   391‚ÜíPipeline:
   392‚Üí1. Upload image ‚Üí stays in browser.
   393‚Üí2. `onnxruntime-web` (WebGPU) runs **Depth Anything V2 Small q4f16** (~18MB) ‚Üí depth map. ~3‚Äì5s on M1 Mac, ~20s on mid hardware.
   394‚Üí3. `@imgly/background-removal` (WASM) ‚Üí subject mask + alpha. ~2‚Äì3s.
   395‚Üí4. Skip LaMa (no good browser inpainting yet); instead use **edge dilation** on the depth + clamp mouse range hard.
   396‚Üí5. R3F shader (Variant A) ‚Äî accept minor stretch artifacts in exchange for zero infra.
   397‚Üí
   398‚Üí**Why #2**: Zero server compute cost, fully private (image never leaves browser). Great for a free tier / demo. Quality ceiling lower due to no real inpainting. Tiefling's open-source MIT code is a direct reference implementation.
   399‚Üí
   400‚Üí### ü•â Rank 3 ‚Äî Cloud API (Immersity AI or Replicate)
   401‚Üí**Quality: 10/10 | Feasibility: 9/10 | Effort: Very Low | Cost: $$**
   402‚Üí
   403‚ÜíPipeline:
   404‚Üí1. Upload ‚Üí POST to Immersity AI API (or `pollinations/3d-photo-inpainting` on Replicate).
   405‚Üí2. Receive: depth map + layered 3D data, OR pre-rendered video.
   406‚Üí3. If interactive: render with their viewer / our shader using their layered output. If video: just `<video autoplay loop muted>`.
   407‚Üí
   408‚Üí**Why #3**: Best quality bar none, least engineering. But: per-image cost ($0.05‚Äì0.50), privacy concerns (user images uploaded to third party), rate limits, and vendor lock-in. Good as a **premium upsell** ("Pro" mode) layered on top of Rank 1.
   409‚Üí
   410‚Üí### Rank 4 ‚Äî Stable Diffusion Inpainting for background
   411‚Üí**Quality: 9/10 | Feasibility: 6/10 | Effort: Medium-High | Cost: Medium**
   412‚Üí
   413‚ÜíLike Rank 1 but replace LaMa with **SD Inpainting** (or SDXL inpaint). Higher quality reconstruction (especially for complex backgrounds like foliage, patterns). But 5‚Äì15s/image on GPU and risk of hallucinated content. Use only for premium tier.
   414‚Üí
   415‚Üí### Rank 5 ‚Äî Hard 3-layer CSS-only parallax (no depth model)
   416‚Üí**Quality: 5/10 | Feasibility: 10/10 | Effort: Very Low | Cost: Free**
   417‚Üí
   418‚ÜíSkip depth model entirely. User (or simple auto-segmentation) provides 3 layers as separate PNGs: sky / midground / subject. Animate with GSAP Variant (d). Cheapest, fastest, but quality limited and requires either manual prep or Rembg + manual layering.

   586‚Üí
   587‚Üí### PART A ‚Äî Depth Slicing & Multi-Layer Decomposition (6‚Äì12 layers)
   588‚Üí
   589‚ÜíWe currently get a depth map (`depthUrl`) and an inpainted background (`backgroundUrl`) from the `/api/separate` route, then the VLM nominates 4 layers by name. To go to 6‚Äì12 layers we need to **slice the depth map algorithmically** and emit a mask + inpainted plate per band.
   590‚Üí
   591‚Üí#### A.1 ‚Äî Depth slicing / quantization (3 concrete algorithms)
   592‚Üí
   593‚ÜíThe depth map from image-edit is a continuous grayscale PNG. Normalise to [0,1] then split into N bands. Three algorithms, ranked by quality:
   594‚Üí
   595‚Üí**(a) Equal-interval slicing** (baseline, naive):
   596‚Üí```
   597‚Üífunction sliceEqual(depthNorm, N) {
   598‚Üí  // depthNorm: Float32Array in [0,1]
   599‚Üí  // returns: masks: boolean[N][H*W]
   600‚Üí  const masks = Array.from({length:N}, () => new Uint8Array(depthNorm.length))
   601‚Üí  for (let i=0; i<depthNorm.length; i++) {
   602‚Üí    const band = Math.min(N-1, Math.floor(depthNorm[i] * N))
   603‚Üí    masks[band][i] = 1
   604‚Üí  }
   605‚Üí  return masks
   606‚Üí}
   607‚Üí```
   608‚Üí- ‚úÖ Trivial, O(N).
   609‚Üí- ‚ùå Over-segments smooth gradients, under-segments clustered scenes (a portrait with a flat sky wastes 8 bands on sky).
   610‚Üí
   611‚Üí**(b) K-means 1D on depth histogram** (recommended for adaptive bands):
   612‚Üí```
   613‚Üífunction sliceKMeans1D(depthNorm, K, iters=20) {
   614‚Üí  // 1. Build 256-bin histogram of depth values
   615‚Üí  const hist = new Float32Array(256)
   616‚Üí  for (const v of depthNorm) hist[Math.min(255, Math.floor(v*255))]++
   617‚Üí  // 2. Init centroids at equal-quantile positions
   618‚Üí  let centroids = quantileInit(hist, K)
   619‚Üí  // 3. Lloyd iterations
   620‚Üí  for (let it=0; it<iters; it++) {
   621‚Üí    const sums = new Float64Array(K), counts = new Float64Array(K)
   622‚Üí    for (let b=0; b<256; b++) {
   623‚Üí      let nearest = 0, best = Infinity
   624‚Üí      for (let k=0; k<K; k++) {
   625‚Üí        const d = Math.abs(b - centroids[k])
   626‚Üí        if (d < best) { best = d; nearest = k }
   627‚Üí      }
   628‚Üí      sums[nearest] += b * hist[b]
   629‚Üí      counts[nearest] += hist[b]
   630‚Üí    }
   631‚Üí    for (let k=0; k<K; k++) if (counts[k] > 0) centroids[k] = sums[k] / counts[k]
   632‚Üí  }
   633‚Üí  // 4. Thresholds = midpoints between sorted centroids
   634‚Üí  centroids.sort((a,b)=>a-b)
   635‚Üí  const thresholds = []
   636‚Üí  for (let k=0; k<K-1; k++) thresholds.push((centroids[k]+centroids[k+1])/2 / 255)
   637‚Üí  return sliceByThresholds(depthNorm, thresholds)
   638‚Üí}
   639‚Üí```
   640‚Üí- ‚úÖ Adapts to actual depth distribution. Better for portraits (sky collapses to 1 band, subject gets 5).
   641‚Üí- ‚úÖ O(256¬∑K¬∑iters + N) ‚Äî extremely fast.
   642‚Üí- This is the algorithm `sklearn.cluster.KMeans(n_clusters=K, n_init=4)` would produce on the 1D depth histogram.
   643‚Üí- Reference: Kaggle "K-Means Clustering - Image Quantization", ML Mastery "K-Means in OpenCV".
   644‚Üí
   645‚Üí**(c) Multi-Otsu thresholding** (best quality, recommended default):
   646‚ÜíFrom scikit-image `threshold_multiotsu` docs (Liao, Chen, Chung 2001):
   647‚Üí```
   648‚Üífunction multiOtsu(depth8, classes=6) {
   649‚Üí  // depth8: Uint8Array 0..255
   650‚Üí  const hist = new Float64Array(256)
   651‚Üí  for (const v of depth8) hist[v]++
   652‚Üí  const total = depth8.length
   653‚Üí  for (let i=0;i<256;i++) hist[i] /= total
   654‚Üí  // Pre-compute cumulative sums P[i] and cumulative means S[i]
   655‚Üí  const P = new Float64Array(256), S = new Float64Array(256)
   656‚Üí  P[0]=hist[0]; S[0]=0
   657‚Üí  for (let i=1;i<256;i++){P[i]=P[i-1]+hist[i]; S[i]=S[i-1]+i*hist[i]}
   658‚Üí  // Maximise between-class variance over all (classes-1) threshold combinations
   659‚Üí  // For classes<=6 this is O(256^(classes-1)) ‚Äî ~10^13 for 6 ‚Äî too slow naively.
   660‚Üí  // Use the Liao et al. fast lookup (precompute H(i,j)=sigma between i,j) ‚Üí O(256^2) preprocessing then O(256^(k-1)) search
   661‚Üí  // Pragmatic: for classes<=4 brute force is fine; for 5-10 use the Liao fast form OR fall back to K-means 1D.
   662‚Üí  // For our app, classes=4‚Äì6: brute force 4-tuple thresholds, with the Liao speedup for >4.
   663‚Üí  ...
   664‚Üí  return thresholds // array of `classes-1` threshold values 0..255
   665‚Üí}
   666‚Üí```
   667‚Üí- ‚úÖ Maximises between-class variance ‚Üí statistically optimal band boundaries.
   668‚Üí- ‚úÖ Default in scikit-image; battle-tested.
   669‚Üí- ‚öÝÔ∏è For >6 classes the brute-force search explodes; use the Liao et al. fast algorithm (precompute `H(i,j)` between-class variance lookup) or fall back to K-means 1D for >6.
   670‚Üí- Reference: `skimage.filters.threshold_multiotsu` (linked above), Liao et al. 2001 "A Fast Algorithm for Multilevel Thresholding".
   671‚Üí
   672‚Üí**Recommendation for our app**: Use **multi-Otsu for 4‚Äì6 layers** (the sweet spot ‚Äî fast + optimal). Switch to **K-means 1D for 7‚Äì12 layers** (O(N) regardless of K). Run server-side in `/api/separate` after image-edit returns the depth map; return an array of mask PNGs (one per layer) alongside the existing `backgroundUrl`.
   673‚Üí
   674‚Üí#### A.2 ‚Äî Layer dilation / edge expansion (no-gap parallax)
   675‚Üí
   676‚ÜíWhen layers translate independently (mouse parallax), the boundary between depth band `k` and band `k+1` produces a transparent gap. Fix: **dilate each layer's mask** so it overlaps its neighbour by a few px.
   677‚Üí
   678‚Üí**Sharp (Node) approach** ‚Äî fastest, already in our project:
   679‚Üí```ts
   680‚Üíimport sharp from 'sharp'
   681‚Üí
   682‚Üíasync function dilateMask(maskPng: Buffer, radiusPx: number): Promise<Buffer> {
   683‚Üí  // morphological dilation via repeated max-filter (3x3 box) `radiusPx` times.
   684‚Üí  // sharp doesn't expose morphology directly; emulate with threshold-blur-threshold:
   685‚Üí  const { data, info } = await sharp(maskPng).raw().toBuffer({resolveWithObject:true})
   686‚Üí  // Cheap trick: blur then threshold ‚Äî produces a "fat" mask.
   687‚Üí  const blurred = await sharp(data, {raw:info})
   688‚Üí    .blur(radiusPx/2 | 0 || 1)
   689‚Üí    .raw().toBuffer()
   690‚Üí  // Threshold at 128: any pixel >128 becomes fully opaque
   691‚Üí  for (let i=0;i<blurred.length;i+=info.channels){
   692‚Üí    const v = blurred[i]
   693‚Üí    const out = v > 64 ? 255 : 0
   694‚Üí    blurred[i]=blurred[i+1]=blurred[i+2]=out
   695‚Üí  }
   696‚Üí  return sharp(blurred, {raw:info}).png().toBuffer()
   697‚Üí}
   698‚Üí```
   699‚Üí
   700‚Üí**True morphological dilation** (use `jimp` or `@napi-rs/canvas`):
   701‚Üí```ts
   702‚Üí// Implement classical dilation: out[p] = max(in[q] for q in 3x3 neighbourhood)
   703‚Üí// Repeat `radiusPx` times. Equivalent to cv2.dilate(mask, kernel=cv2.ones(3,3), iterations=r)
   704‚Üífunction dilate(mask: Uint8Array, w: number, h: number, radius: number): Uint8Array {
   705‚Üí  let cur = mask.slice()
   706‚Üí  for (let it=0; it<radius; it++) {
   707‚Üí    const next = new Uint8Array(cur.length)
   708‚Üí    for (let y=0;y<h;y++) for (let x=0;x<w;x++){
   709‚Üí      let m=0
   710‚Üí      for (let dy=-1;dy<=1;dy++) for (let dx=-1;dx<=1;dx++){
   711‚Üí        const nx=x+dx, ny=y+dy
   712‚Üí        if (nx<0||ny<0||nx>=w||ny>=h) continue
   713‚Üí        m = Math.max(m, cur[ny*w+nx])
   714‚Üí      }
   715‚Üí      next[y*w+x] = m
   716‚Üí    }
   717‚Üí    cur = next
   718‚Üí  }
   719‚Üí  return cur
   720‚Üí}
   721‚Üí```
   722‚Üí- **Recommended radius**: 10‚Äì30 px (Tiefling uses `expandDepthmapRadius‚âà25`; Facebook 3D Photos use similar). Scale with target resolution: `radius = Math.round(maxDim * 0.02)`.
   723‚Üí- Always dilate **after** slicing, **before** inpainting (so the inpainted plate covers the dilated band).
   724‚Üí- Reference: PyImageSearch "OpenCV Morphological Operations", Tiefling `expandDepthmapRadius` (cited in Task 1).
   725‚Üí
   726‚Üí#### A.3 ‚Äî Per-layer background inpainting
   727‚Üí
   728‚ÜíThree strategies, ranked by quality/effort for our sandbox (VLM + image-edit only, no LaMa/SD):
   729‚Üí
   730‚Üí**(a) Reuse the single inpainted back plate** (cheapest, already implemented):
   731‚Üí- We already have `backgroundUrl` (subject removed). Use it as the universal back plate. Each layer above shows through where its (dilated) mask is empty.
   732‚Üí- ‚úÖ Zero extra compute. Works fine for 4‚Äì6 layers.
   733‚Üí- ‚ùå For 8‚Äì12 layers, mid-depth bands need their *own* occluded content (e.g. a tree in front of mountains in front of sky ‚Äî removing the tree shouldn't show the sky directly).
   734‚Üí
   735‚Üí**(b) "Fill from neighboring depth band" trick** (no ML, fast):
   736‚Üí- For each layer `k`, the occluded pixels are filled by **stretching the next-farther layer's color** into the gap. Like "content-aware extend" via `sharp.resize` with `fit:'fill'` + edge mirroring.
   737‚Üí- ‚úÖ No ML. Fast. Looks plausible for natural scenes.
   738‚Üí- ‚ùå Fails on textured regions (repeats patterns visibly).
   739‚Üí
   740‚Üí**(c) Per-layer VLM re-prompt to image-edit** (the sandbox-native approach):
   741‚Üí- For each depth band `k` with name `layerName` (from VLM analysis), re-prompt z-ai image-edit: *"Show what's behind the {layerName} in this image. Same lighting, same style."*
   742‚Üí- ‚úÖ Uses tools we already have. Each layer gets its own occluded-content plate.
   743‚Üí- ‚öÝÔ∏è 6‚Äì12 image-edit calls per project (~3‚Äì4s each = 20‚Äì50s total). Mitigate with `Promise.all` parallelism + caching.
   744‚Üí- This is the recommended upgrade path for v2: server `/api/separate` returns `{ layers: [{id, name, depth, maskUrl, plateUrl}] }` where each `plateUrl` is the inpainted content for that band.
   745‚Üí
   746‚Üí**(d) LaMa via IOPaint** (gold standard, requires Docker ‚Äî out of sandbox):
   747‚Üí- Already noted in Task 1: `POST http://iopaint:8000/inpaint {image, mask}` ‚Üí PNG. ~1s/image on CPU. Use this if we move off the sandbox.
   748‚Üí
   749‚Üí#### A.4 ‚Äî Order-independent compositing (painter's algorithm with z-sort)
   750‚Üí
   751‚ÜíFrom GeeksforGeeks "Painter's Algorithm":
   752‚Üí> "Sort polygons by depth (decreasing z = farthest first). Paint each in order, nearer polygons overwrite farther ones."
   753‚Üí
   754‚ÜíFor our flat depth-banded layers (each a textured plane), this is trivial:
   755‚Üí```ts
   756‚Üí// sort back-to-front (farthest depth = smallest depth value, where depth=0 is sky, depth=1 is foreground)
   757‚Üíconst sortedLayers = [...layers].sort((a,b) => a.depth - b.depth)
   758‚Üí// render in order; later renders cover earlier where masks overlap
   759‚Üífor (const layer of sortedLayers) ctx.drawImage(layer.canvas, 0, 0)
   760‚Üí```
   761‚Üí- For non-intersecting planes this is **always correct** (no need for BSP/z-buffer).
   762‚Üí- For our scene (depth-banded layers with dilated masks) planes never intersect ‚Üí painter's algorithm is sufficient.
   763‚Üí- If we ever add rotated/skewed layers, fall back to per-pixel z-test in the WebGL shader (already supported by depth buffer).
   764‚Üí
   765‚Üí#### A.5 ‚Äî Adaptive meshing (vertex-displaced mesh per depth band)
   766‚Üí
   767‚ÜíCurrently `AliveWebGL.tsx` uses a single fullscreen quad with UV-offset. Upgrading to **vertex-displaced mesh** gives true parallax geometry and avoids rubber-band stretch.
   768‚Üí
   769‚Üí**Codrops "Dynamic Terrain Deformation with R3F" pattern** (read in full):
   770‚Üí```tsx
   771‚Üí<mesh rotation={[-Math.PI/2, 0, 0]}>
   772‚Üí  <planeGeometry args={[W, H, GRID_RESOLUTION, GRID_RESOLUTION]} />
   773‚Üí  <meshStandardMaterial
   774‚Üí    map={colorMap}
   775‚Üí    displacementMap={depthMap}
   776‚Üí    displacementScale={2}
   777‚Üí  />
   778‚Üí</mesh>
   779‚Üí```
   780‚Üí- `GRID_RESOLUTION` = 128 or 256 subdivisions per axis (gives 65K‚Äì65K vertices ‚Äî fine for desktop GPUs).
   781‚Üí- Per-vertex Z comes directly from the displacementMap; Three.js handles the GPU upload.
   782‚Üí
   783‚Üí**For multi-layer (6‚Äì12 layers)** ‚Äî generate one mesh per depth band:
   784‚Üí```tsx
   785‚ÜísortedLayers.map(layer => (
   786‚Üí  <mesh key={layer.id} position={[0, 0, layer.depth * DEPTH_SCALE]}>
   787‚Üí    <planeGeometry args={[W, H, GRID, GRID]} />
   788‚Üí    <meshBasicMaterial
   789‚Üí      map={layer.texture}
   790‚Üí      alphaMap={layer.maskTexture}  // the dilated band mask
   791‚Üí      transparent
   792‚Üí      depthWrite={true}
   793‚Üí    />
   794‚Üí    {/* Per-vertex Z from band-local depth (cropped from full depth map) */}
   795‚Üí    <displacementMap ... />
   796‚Üí  </mesh>
   797‚Üí))
   798‚Üí```
   799‚Üí- Use `THREE.PlaneGeometry` (not BufferGeometry) ‚Äî `displacementMap` works out of the box.
   800‚Üí- For 8 layers √ó 128¬≤ verts = 130K verts total ‚Äî modern GPUs handle this easily at 60fps.
   801‚Üí
   802‚Üí**Adaptive meshing (Tiefling technique)** ‚Äî subdivide more at depth discontinuities:
   803‚Üí- Compute Sobel edge magnitude of the depth map; where `sobel > threshold`, add 2‚Äì4√ó more subdivisions.
   804‚Üí- Implementation: build a custom BufferGeometry where vertex density is high at edges, low in flat regions. ~5K verts total for typical scene (vs 65K uniform).
   805‚Üí- ‚úÖ 10√ó fewer verts, same visual quality.
   806‚Üí- ‚ùå More complex; defer to v3.

   946‚Üí```ts
   947‚Üíconst degToRad = (a) => (a/180)*Math.PI
   948‚Üíconst getCorner = (px, py, dx, dy, angle) => {
   949‚Üí  const dist = Math.sqrt(dx*dx + dy*dy)
   950‚Üí  angle += Math.atan2(dy, dx)
   951‚Üí  return {x: px + dist*Math.cos(angle), y: py + dist*Math.sin(angle)}
   952‚Üí}
   953‚Üíconst getClientRect = (el) => {
   954‚Üí  const {x, y, width, height, rotation=0} = el
   955‚Üí  const rad = degToRad(rotation)
   956‚Üí  const corners = [
   957‚Üí    getCorner(x,y, 0,0, rad),
   958‚Üí    getCorner(x,y, width,0, rad),
   959‚Üí    getCorner(x,y, width,height, rad),
   960‚Üí    getCorner(x,y, 0,height, rad),
   961‚Üí  ]
   962‚Üí  const xs = corners.map(c=>c.x), ys = corners.map(c=>c.y)
   963‚Üí  return {x:Math.min(...xs), y:Math.min(...ys), width:Math.max(...xs)-Math.min(...xs), height:Math.max(...ys)-Math.min(...ys)}
   964‚Üí}
   965‚Üí// Click hit-test:
   966‚Üíconst isHit = (mousePos, layer) => {
   967‚Üí  // inverse-rotate mouse into layer local space, then AABB test
   968‚Üí  const rad = degToRad(-layer.rotation)
   969‚Üí  const dx = mousePos.x - (layer.x + layer.width/2)
   970‚Üí  const dy = mousePos.y - (layer.y + layer.height/2)
   971‚Üí  const lx = dx*Math.cos(rad) - dy*Math.sin(rad) + layer.width/2
   972‚Üí  const ly = dx*Math.sin(rad) + dy*Math.cos(rad) + layer.height/2
   973‚Üí  return lx >= 0 && lx <= layer.width && ly >= 0 && ly <= layer.height
   974‚Üí}
   975‚Üí```

  1053‚Üí#### C.2 ‚Äî Jitter / boil (Spider-Verse "boiling" effect)
  1054‚Üí
  1055‚ÜíFrom Camillo Visini's tutorial (read in full ‚Äî the canonical web implementation):
  1056‚Üí```html
  1057‚Üí<svg width="0" height="0" style="position:absolute">
  1058‚Üí  <filter id="boil">
  1059‚Üí    <feTurbulence type="turbulence" baseFrequency="0.012" numOctaves="2" seed="1" result="noise"/>
  1060‚Üí    <feDisplacementMap in="SourceGraphic" in2="noise" scale="4"
  1061‚Üí      xChannelSelector="R" yChannelSelector="G"/>
  1062‚Üí  </filter>
  1063‚Üí</svg>
  1064‚Üí<img src="layer.png" style="filter:url(#boil)" id="boilTarget"/>
  1065‚Üí```
  1066‚Üí```ts
  1067‚Üí// Animate by cycling baseFrequency every 100ms with small offsets (the "boil" loop)
  1068‚Üíconst offsets = [-0.002, 0.0014, -0.0011, 0.0021, -0.0017]
  1069‚Üílet i = 0
  1070‚Üíconst baseFreq = 0.012
  1071‚Üíconst scale = 0.6  // 0..1, "animation scale" ‚Äî Visini's term
  1072‚ÜísetInterval(() => {
  1073‚Üí  const offset = offsets[i % offsets.length] * scale
  1074‚Üí  const turb = document.querySelector('#boil feTurbulence')
  1075‚Üí  turb.setAttribute('baseFrequency', (baseFreq + offset).toFixed(5))
  1076‚Üí  i++
  1077‚Üí}, 100)
  1078‚Üí```
  1079‚Üí**Key insight from Visini**: don't animate `seed` (causes hard jumps); animate `baseFrequency` by tiny offsets (¬±0.002) every 100ms. This produces the "hand-drawn wobble" loop. Use prime period (e.g. 4 or 5 offsets) so it never visually repeats.
  1080‚Üí- **Per-layer**: each layer gets its own `<filter id="boil-{layerId}">` with different `baseFrequency` and offset array ‚Üí layers boil out of sync.
  1081‚Üí- Recommended `scale` values: 0.3 (subtle), 0.8 (visible), 1.5 (extreme, "Into the Spider-Verse" look).

  1083‚Üí#### C.3 ‚Äî Wave distortion (horizontal sinusoidal displacement)
  1084‚Üí
  1085‚ÜíDifferent from liquid (which is 2D turbulence). Wave is 1D horizontal sine ‚Äî like a flag rippling.
  1086‚Üí
  1087‚Üí```glsl
  1088‚Üí// In our existing AliveWebGL fragment shader, add:
  1089‚Üíuniform float uWaveAmp;     // ~0.005 UV
  1090‚Üíuniform float uWaveFreq;    // ~12.0 cycles across image
  1091‚Üíuniform float uWaveTime;
  1092‚Üívarying vec2 vUv;
  1093‚Üí
  1094‚Üívoid main() {
  1095‚Üí  float wave = sin(vUv.x * uWaveFreq + uWaveTime * 1.6) * uWaveAmp;
  1096‚Üí  vec2 uv = vUv + vec2(0.0, wave);  // vertical displacement from horizontal sine
  1097‚Üí  gl_FragColor = texture2D(uImage, uv);
  1098‚Üí}
  1099‚Üí```

  1131‚Üí#### C.5 ‚Äî Hue drift (very slow hue-rotate, 5‚Äì10¬∞ over 30s)

  1140‚Üí  filter: hue-rotate(var(--hue));
  1141‚Üí}
  1142‚Üí@keyframes hueDrift {
  1143‚Üí  from { --hue: 0deg; }
  1144‚Üí  to   { --hue: 8deg; }

  1222‚Üí#### C.9 ‚Äî Mouse velocity influence (layers react to mouse SPEED not just position)
  1223‚Üí
  1224‚ÜíCurrently parallax follows `mousePos`. Velocity adds inertia: when mouse moves fast, layers "lag" and overshoot.
  1225‚Üí
  1226‚Üí```ts
  1227‚Üíconst mouse = {x:0, y:0, px:0, py:0, vx:0, vy:0}
  1228‚Üíwindow.addEventListener('pointermove', (e) => {
  1229‚Üí  mouse.x = e.clientX / window.innerWidth - 0.5
  1230‚Üí  mouse.y = e.clientY / window.innerHeight - 0.5
  1231‚Üí})
  1232‚Üí// In rAF loop:
  1233‚Üífunction tick() {
  1234‚Üí  // velocity = (current - previous) / dt
  1235‚Üí  mouse.vx = (mouse.x - mouse.px) * 60  // assume 60fps
  1236‚Üí  mouse.vy = (mouse.y - mouse.py) * 60
  1237‚Üí  mouse.px = mouse.x
  1238‚Üí  mouse.py = mouse.y
  1239‚Üí  // Low-pass filter (smoothing)
  1240‚Üí  const smoothVx = lerp(state.svx, mouse.vx, 0.08)
  1241‚Üí  const smoothVy = lerp(state.svy, mouse.vy, 0.08)
  1242‚Üí  // Apply: position = mousePos * depth + velocity * depth * inertia
  1243‚Üí  layers.forEach(layer => {
  1244‚Üí    const tx = mouse.x * layer.depth * 40 + smoothVx * layer.depth * 80
  1245‚Üí    const ty = mouse.y * layer.depth * 40 + smoothVy * layer.depth * 80
  1246‚Üí    gsap.set(layer.el, {x:tx, y:ty})
  1247‚Üí  })
  1248‚Üí  requestAnimationFrame(tick)
  1249‚Üí}
  1250‚Üí```
  1251‚Üí- Velocity amplitude (80) is 2√ó position amplitude (40) ‚Üí fast moves dominate.
  1252‚Üí- Low-pass filter (0.08) prevents jitter on slow moves.
  1253‚Üí
  1254‚Üí#### C.10 ‚Äî Spring physics per layer (independent stiffness/damping/mass)
  1255‚Üí
  1256‚ÜíFrom Maxime Heckel's article (read in full ‚Äî the canonical explanation). Spring physics math:
  1257‚Üí```
  1258‚ÜíFspring = -k * (x - restLength)        // Hooke's law
  1259‚ÜíFdamping = -d * v                      // damping force
  1260‚Üía = (Fspring + Fdamping) / m           // Newton's 2nd law
  1261‚Üív += a * dt                            // integrate velocity
  1262‚Üíx += v * dt                            // integrate position
  1263‚Üí```
  1264‚ÜíJavaScript implementation (one layer):
  1265‚Üí```ts
  1266‚Üíclass Spring {
  1267‚Üí  constructor({stiffness=100, damping=10, mass=1, rest=0} = {}) {
  1268‚Üí    this.k = stiffness; this.d = damping; this.m = mass
  1269‚Üí    this.rest = rest; this.x = rest; this.v = 0
  1270‚Üí  }
  1271‚Üí  setTarget(t) { this.rest = t }
  1272‚Üí  tick(dt = 1/60) {
  1273‚Üí    const Fspring = -this.k * (this.x - this.rest)
  1274‚Üí    const Fdamping = -this.d * this.v
  1275‚Üí    const a = (Fspring + Fdamping) / this.m
  1276‚Üí    this.v += a * dt
  1277‚Üí    this.x += this.v * dt
  1278‚Üí    return this.x
  1279‚Üí  }
  1280‚Üí}
  1281‚Üí```
  1282‚Üí- **Per-layer config**: add `springStiffness`, `springDamping`, `springMass` to `LayerAnimationConfig`. Default (Framer's defaults): stiffness=100, damping=10, mass=1.
  1283‚Üí- **Tuning guide**: lower stiffness = looser/slower; higher damping = less overshoot; higher mass = sluggish.
  1284‚Üí- **Use Framer Motion's `useSpring`** for declarative version:
  1285‚Üí```tsx
  1286‚Üíconst x = useSpring(0, {stiffness: layer.springStiffness, damping: layer.springDamping, mass: layer.springMass})
  1287‚ÜíuseEffect(() => x.set(mouseX * depth * 40), [mouseX])
  1288‚Üíreturn <motion.div style={{x}} />
  1289‚Üí```

  1291‚Üí#### C.11 ‚Äî Phase offset between layers (staggered animation start so layers never sync)
  1292‚Üí
  1293‚ÜíFrom motion.dev `stagger()` docs (read in full):
  1294‚Üí```ts
  1295‚Üíimport { animate, stagger } from 'motion'
  1296‚Üí
  1297‚Üí// Stagger all layers' entrance by 80ms each
  1298‚Üíanimate('.alive-layer', {opacity: [0, 1]}, {delay: stagger(0.08)})
  1299‚Üí
  1300‚Üí// Stagger from center outward
  1301‚Üíanimate('.alive-layer', {scale: [0.9, 1]}, {delay: stagger(0.05, {from: 'center'})})
  1302‚Üí
  1303‚Üí// With easing for non-linear stagger
  1304‚Üíanimate('.alive-layer', {y: [20, 0]}, {delay: stagger(0.1, {ease: 'easeOut'})})
  1305‚Üí
  1306‚Üí// Negative startDelay = layers start mid-animation (never sync)
  1307‚Üíanimate('.alive-layer', {rotate: [0, 360]}, {
  1308‚Üí  duration: 20, repeat: Infinity, ease: 'linear',
  1309‚Üí  delay: stagger(0.7, {startDelay: -2})  // each layer starts 0.7s into the loop, offset
  1310‚Üí})
  1311‚Üí```
  1312‚Üí- `stagger()` options: `startDelay`, `from` ("first"|"center"|"last"|index), `ease`.
  1313‚Üí- **For continuous loops** (breathing, sway), the cleanest pattern: each layer's CSS animation has a `animation-delay` of `-1.3s * layerDepth` (negative = starts mid-cycle) ‚Üí layers never visually sync. Use prime offsets (1.3, 2.7, 4.1‚Ä¶) to maximise desync.
  1314‚Üí- Reference: CSS-Tricks "Different Approaches for Creating a Staggered Animation".
  1315‚Üí
  1316‚Üí#### C.12 ‚Äî Simplex noise per-layer motion (curl noise displacement)
  1317‚Üí
  1318‚Üí```bash
  1319‚Üínpm install simplex-noise@4
  1320‚Üí```
  1321‚Üí```ts
  1322‚Üíimport { createNoise2D } from 'simplex-noise'
  1323‚Üí
  1324‚Üíconst noise2D = createNoise2D(() => Math.random())
  1325‚Üí// Per-layer noise field seeded differently:
  1326‚Üíconst layerNoise = createNoise2D(() => layer.seed * 1337)
  1327‚Üí
  1328‚Üífunction tick(dt: number, t: number) {
  1329‚Üí  for (const layer of layers) {
  1330‚Üí    // Sample 2 noise points slightly apart ‚Üí "curl" gives divergence-free field
  1331‚Üí    const eps = 0.001
  1332‚Üí    const n1 = layerNoise(t * 0.05,            layer.seed)
  1333‚Üí    const n2 = layerNoise(t * 0.05 + eps,      layer.seed)
  1334‚Üí    const n3 = layerNoise(layer.seed,          t * 0.05)
  1335‚Üí    const n4 = layerNoise(layer.seed + eps,    t * 0.05)
  1336‚Üí    const dx = (n2 - n1) / eps * 20  // scale to px
  1337‚Üí    const dy = (n4 - n3) / eps * 20
  1338‚Üí    layer.el.style.transform = `translate(${dx}px, ${dy}px)`
  1339‚Üí  }
  1340‚Üí  requestAnimationFrame((nt) => tick(dt, nt / 1000))
  1341‚Üí}
  1342‚Üí```
  1343‚Üí- **Curl noise** (Bridson et al.): gives smooth divergence-free velocity fields ‚Üí particles/layers flow without clumping.
  1344‚Üí- **Per-layer**: different `seed` (or different noise function instance) per layer ‚Üí independent motion.
  1345‚Üí- Amplitude: ~10‚Äì30px. Time scale: `0.02‚Äì0.1` (slow drift).

  1406‚Üí#### D.3 ‚Äî Light rays / god rays (radial blur post-process)
  1407‚Üí
  1408‚ÜíFrom Andrew Berg's article (Kenny Mitchell, GPU Gems 3 ‚Äî read in full):
  1409‚Üí
  1410‚Üí**The technique** (3 steps):
  1411‚Üí1. Render the scene with light source = white, occluders = black ‚Üí "occlusion texture".
  1412‚Üí2. Apply radial blur toward the light source in a fragment shader:
  1413‚Üí```glsl
  1414‚Üí// Volumetric light scattering shader (Mitchell, GPU Gems 3)
  1415‚Üíuniform sampler2D uOcclusion;
  1416‚Üíuniform vec2 uLightPos;       // screen-space light position
  1417‚Üíuniform float uDecay;         // 0.95
  1418‚Üíuniform float uDensity;       // 0.5
  1419‚Üíuniform float uWeight;        // 0.4
  1420‚Üíuniform float uExposure;      // 0.2
  1421‚Üíuniform int uSamples;         // 80‚Äì100
  1422‚Üí
  1423‚Üívoid main() {
  1424‚Üí  vec2 uv = vUv;
  1425‚Üí  vec2 dir = uv - uLightPos;
  1426‚Üí  dir *= uDensity;  // step size
  1427‚Üí  float illum = 1.0;
  1428‚Üí  vec4 col = texture2D(uOcclusion, uv);
  1429‚Üí  for (int i=0; i<100; i++) {
  1430‚Üí    if (i >= uSamples) break;
  1431‚Üí    uv -= dir;
  1432‚Üí    vec4 c = texture2D(uOcclusion, uv);
  1433‚Üí    c *= illum * uWeight;
  1434‚Üí    col += c;
  1435‚Üí    illum *= uDecay;
  1436‚Üí  }
  1437‚Üí  gl_FragColor = col * uExposure;
  1438‚Üí}
  1439‚Üí```
  1440‚Üí3. Additively blend the result over the original scene render.
  1441‚Üí
  1442‚Üí**Cheaper pseudo-volumetric** (three.jsdemos.com pattern ‚Äî for our app):
  1443‚Üí- Use a `ConeGeometry` mesh, `MeshBasicMaterial` with `transparent:true, opacity:0.15, blending:AdditiveBlending, side:DoubleSide, depthWrite:false`.
  1444‚Üí- Position the cone tip at the light source, expand toward the scene.
  1445‚Üí- Animate opacity via a sine wave (slow pulse).

  1574‚Üí#### E.2 ‚Äî CSS `@keyframes` vs Web Animations API
  1575‚Üí
  1576‚ÜíFrom CSS-Tricks "Comparison of Animation Technologies" (read in full) + MDN "CSS and JavaScript animation performance":
  1577‚Üí
  1578‚Üí| Use case | CSS `@keyframes` | Web Animations API (WAAPI) | Motion / GSAP |
  1579‚Üí|---|---|---|---|
  1580‚Üí| Simple infinite loops (breathing, sway, hue drift) | ‚úÖ Best ‚Äî declarative, GPU-accelerated, pauses cleanly | OK | OK |
  1581‚Üí| Imperative / scrubbable / one-shot | ‚ùå | ‚úÖ Best ‚Äî `animate()` returns Animation object, can `pause()`/`seek`/`reverse` | ‚úÖ |
  1582‚Üí| Spring physics | ‚ùå | ‚ùå (no native spring) | ‚úÖ (Motion `useSpring`, GSAP `CustomEase`) |
  1583‚Üí| Sequencing (timeline) | ‚ùå Painful with `animation-delay` | OK with `Promise.all` of `.finished` | ‚úÖ Best (GSAP `timeline()`, Motion `animate([segments])`) |
  1584‚Üí| Stagger | ‚ùå Manual `nth-child` | OK via JS | ‚úÖ Motion `stagger()`, GSAP `stagger` config |
  1585‚Üí| Performance | ‚úÖ Best ‚Äî compositor thread | ‚úÖ Same engine as CSS, same perf | ‚úÖ Same |
  1586‚Üí
  1587‚Üí**Recommendation for our app**:
  1588‚Üí- **CSS `@keyframes` + `@property`** for all the infinite subtle loops (breathing, sway, twist, glow, hue drift, focus pull, shadow drift, boil). These run on the compositor thread, don't need JS, and pause cleanly on tab-inactive.
  1589‚Üí- **Motion's `animate()`** (which wraps WAAPI) for: entrance/exit, mouse-driven parallax (imperative), timeline playback, layer reordering transitions.
  1590‚Üí- **GSAP** (already in our project per Task 1) for: complex orchestrated reveals, scroll-triggered sequences. Skip if not needed ‚Äî Motion covers most cases.

  1928‚Üí
  1929‚Üí### A.1 How award-winning studios structure layer systems
  1930‚Üí
  1931‚Üí**Active Theory** (Baillie Gifford, Disney, Google Cube, Marvel) ‚Äî they publish extensive dev case studies. Their layer model:
  1932‚Üí- A `LayerStack` is an array of `Plane` objects in a single WebGL scene. Each `Plane` owns: texture (color+alpha), depth value, parallaxStrength, transform {x, y, scale, rotation}, visible bool, blend mode.
  1933‚Üí- Visibility is enforced at the **scene-graph level** ‚Äî `if (!layer.visible) return;` at the top of the render loop. Invisible layers never reach the GPU. This is the same pattern Pixi.js v8 uses with `container.removeChild` vs `child.visible = false` (the latter short-circuits the render pass).
  1934‚Üí- Transforms are baked into a per-plane `mat4` that the user edits via a transform-gizmo (their in-house editor). The gizmo writes directly to the layer's model matrix on `pointermove` ‚Äî no React reconciliation in the hot path. **This is critical: award-winning editors keep transform state in a mutable ref / WebGL uniform, NOT in React state, to avoid the 16ms React reconciliation cost on every drag frame.**
  1935‚Üí
  1936‚Üí**Lusion** (Awwwards SOTD √ó many, e.g. Atlas Earth, Hume) ‚Äî same pattern: Three.js scene with `Group` per layer, `mesh.visible` flag, transforms on `mesh.position/rotation/scale`. They use GSAP timelines synced to scroll position via ScrollTrigger. Visibility = `mesh.visible = false` (skips frustum culling AND render).
  1937‚Üí
  1938‚Üí**Resn** (Apple, Nike, NASA) ‚Äî tends to use a custom WebGL framework (Wonderland, Babylon, or their own "Resn Engine"). Layer concept is the same: array of `Layer` objects with `visible` and `transform`, but they lean heavily on **per-layer RenderTargets** ‚Äî each layer renders into its own FBO, then composites in a final pass. This lets them apply post-processing (bloom, DOF, chromatic aberration) per-layer without re-running the whole pipeline.
  1939‚Üí
  1940‚Üí**Studio Korpi**, **Mathieu Triay**, **Locomotive** (the eponymous smooth-scroll lib + Locomotive Agency) ‚Äî most use plain DOM + CSS 3D + GSAP. Their layer model is one `<div class="layer" data-depth="0.7">` per layer, with `transform: translate3d(...) translateZ(...)`. `data-depth` drives parallax on scroll/mouse. Visibility via `display: none` (NOT opacity ‚Äî display:none removes from layout so blend modes don't pick up an invisible layer's contribution).
  1941‚Üí
  1942‚Üí**Immersity (ex-LeiaPix)** ‚Äî the closest commercial analog to what we're building. They use a Layered Depth Image (LDI): RGB-D per layer with inpainted occluded regions. Each layer is a textured plane in a Three.js scene, displaced by a per-vertex depth. Visibility toggles skip the plane in the render call. Their public API exposes `layer.visible`, `layer.opacity`, `layer.position3d`, `layer.scale3d`.

  1956‚Üí### A.3 The "alive" image aesthetic on Awwwards ‚Äî what makes stills breathe
  1957‚Üí
  1958‚ÜíFrom the Task 1 worklog + research_cache, the techniques that consistently appear in Awwwards SOTD "alive image" pieces:
  1959‚Üí
  1960‚Üí1. **Mouse / gyro parallax with per-layer depth multiplier** ‚Äî universal. Smoothed via lerp/spring (GSAP `quickTo`, Motion `useSpring`, Lenis damp).
  1961‚Üí2. **Lenis smooth-scroll bound to GSAP ScrollTrigger** ‚Äî for scroll-bound parallax. We currently lack this (mouse only).
  1962‚Üí3. **WebGL displacement** ‚Äî Codrops-style `uv + mouse * depth.r * strength` fragment shader, OR R3F vertex displacement with `displacementMap`. Both well-represented in SOTDs.
  1963‚Üí4. **SVG `feTurbulence` + `feDisplacementMap`** animated via `seed`/`baseFrequency` for liquid shimmer ‚Äî see Codrops "Liquid Distortion Effects" (highly awarded).
  1964‚Üí5. **`@property`-registered CSS custom properties** for animatable filter/transform amplitudes ‚Äî modern SOTDs (2023+) use this to animate `--blur`, `--hue`, `--scale` via plain `@keyframes` instead of JS. **Our v2 already does this** (globals.css has 7 `@property` declarations) ‚Äî keep.
  1965‚Üí6. **Per-layer `mix-blend-mode`** for atmospheric compositing (screen for light leaks, multiply for shadows, soft-light for grain). MDN doc covered in `research_cache/b7_blend.json`.
  1966‚Üí7. **Constrained camera** ‚Äî Google Cinematic Photos team's rule: ‚â§¬±10¬∞ rotation, ‚â§tens-of-px translation. Hides depth-edge artifacts without per-frame inpainting.
  1967‚Üí8. **Phase desync via prime-duration `animation-delay`** ‚Äî already in our v2.
  1968‚Üí9. **Painter's algorithm for z-order** ‚Äî covered in `research_cache/a6_painters.json` + `read_painters.json`: sort by depth back‚Üífront, render in order. We do this via `zIndex: 10 + index + Math.round(depth * 100)`.

  1971‚Üí### A.4 Common pitfalls that break layer editors (from research_cache + studio post-mortems)
  1972‚Üí
  1973‚Üí- **z-index conflicts** ‚Äî when blend modes are active, the DOM z-index AND the painter's-order render order must agree, or you get a "ghost" of an invisible layer bleeding through (because `mix-blend-mode` still samples the layer below it even if it's visually behind). **Fix: when a layer is hidden, set `display: none` so it's removed from the blend stack entirely.** This is exactly our Bug #1.
  1974‚Üí- **transform-origin drift** ‚Äî when scaling/rotating, if `transform-origin` isn't `50% 50% 0` consistently across all layers, the parallax math breaks. Our `.alive-layer` utility uses `inset-0` so origin defaults to center ‚Äî OK.
  1975‚Üí- **`mix-blend-mode` requires an isolated stacking context** ‚Äî if the parent doesn't have `isolation: isolate`, blend modes leak into the page background. Our `AliveStage` root has `bg-black` but no explicit `isolation: isolate`. Worth adding.
  1976‚Üí- **pointer-events blocking parallax** ‚Äî a layer with `pointer-events: auto` captures `pointermove` and stops it reaching the container's listener. Our code sets `pointerEvents: editorMode ? "auto" : "none"` per plane (AliveLayers.tsx:328) ‚Äî correct, but in editor mode ALL planes are `auto`, so the topmost plane gets all events and lower planes can't be selected. Active Theory's fix: only the SELECTED plane has `pointer-events: auto`; siblings are `none`. **We don't do this.**
  1977‚Üí- **react-moveable + framer-motion transform fight** ‚Äî known issue. moveable writes inline `transform` on the target; framer-motion's `style.x/y/scale/rotate` reconciliation overwrites it. The correct integration is to either (a) NOT use framer-motion's transform shortcuts on moveable's target, (b) use a separate wrapper element where moveable manipulates one and framer-motion manipulates the other, or (c) drive moveable's transform externally via its `transform` prop and lock the target. **Our LayerEditor + AliveLayers combination hits pitfall (a)** ‚Äî see Bug #2.
  1978‚Üí- **react-moveable `beforeTranslate` semantics** ‚Äî `OnDrag.beforeTranslate` is the **cumulative delta from drag start**, NOT the per-frame delta. If you do `setX(layer.x + beforeTranslate[0])` on every event, you double-count because `layer.x` is also being updated each frame. The correct pattern is: capture `layer.x`/`layer.y` on `onDragStart` into a ref, then `setX(dragStartRef.x + beforeTranslate[0])`. **Our LayerEditor does exactly the wrong thing** ‚Äî see Bug #2.
  1979‚Üí- **`useTransform(value, fn)` closure staleness** ‚Äî Framer Motion's `useTransform(input, transformer)` only re-emits when `input` emits. If `transformer` captures external state (like `t.x`), that state updates silently without re-running the transformer. The correct pattern is to pass ALL inputs as MotionValues: `useTransform([smx, tXMV], ([v, tx]) => v * pxToMove + tx)`. **Our AliveLayers does exactly the wrong thing** ‚Äî see Bug #2.
